# Cost Simulation System - Monorepo Organization Recommendation

**Date**: 2026-01-13
**Status**: Structural Design
**Question**: Should Cost Simulation be in a subdirectory within VelocityBench?
**Answer**: YES - and here's the optimal structure

---

## Current VelocityBench Structure

```
velocitybench/
├── database/              # PostgreSQL schema + seed data
├── frameworks/            # 48+ framework implementations
├── monitoring/            # Prometheus, Grafana, cAdvisor
├── tests/                 # Integration, performance, QA tests
├── docs/                  # Documentation
├── scripts/               # Orchestration scripts (run-benchmarks.py, etc)
├── phase-plans/           # Implementation roadmaps
├── testing-templates/     # Reusable test patterns
└── .github/               # CI/CD workflows
```

**This is a monorepo**: One repo contains multiple related subsystems (frameworks, database, tests, monitoring)

---

## Why Cost Simulation Should Be a Subdirectory

### ✅ Architectural Alignment

**Cost Simulation is a peer component to existing systems:**

```
VelocityBench = Data Collection + Test Execution + Analysis + Monitoring + Cost Analysis

                      ↓

velocitybench/
├── database/              # Data Collection (PostgreSQL schema)
├── frameworks/            # Test Execution (48+ framework impls)
├── tests/                 # Test Execution (JMeter, integration tests)
├── monitoring/            # Analysis (Prometheus, Grafana, metrics)
├── costs/                 # Analysis (Cost simulation, efficiency)  ← NEW
└── scripts/               # Orchestration (tie everything together)
```

**Cost Simulation fits naturally as an analysis layer alongside monitoring**

### ✅ Monorepo Best Practices

1. **Related but independent**: Cost simulation works independently but integrates with benchmarks
2. **Shared dependencies**: Uses same Python, same results directory structure, same framework config
3. **Single deployment**: Ships as part of VelocityBench, not separate tool
4. **Clear ownership**: Easy to find, understand, and maintain
5. **Cross-component integration**: Hooks into scripts, monitoring, results directory

### ✅ Project Maturity

VelocityBench is already structured as a monorepo:
- `database/` - one concern
- `frameworks/` - another concern
- `monitoring/` - another concern
- `tests/` - another concern
- `scripts/` - orchestration

**Cost simulation is the natural 5th concern**

## **RECOMMENDATION: Option 1 (`costs/` directory)**

### Why Option 1 is Best

✅ **Clear Organization**
- Parallel to `monitoring/`, `database/`, `frameworks/`, `tests/`
- Easy to understand where things go
- Mirrors the monorepo structure

✅ **Low Coupling**
- Cost simulation doesn't require monitoring infrastructure
- Stands alone, but integrates via results directory
- Could theoretically be extracted later if needed

✅ **Scalability**
- If in future you add cost simulation for different cloud providers or models
- Easy to add `costs/models/`, `costs/providers/`, etc.
- Clean namespace growth

✅ **Discoverability**
- Users expect to find all analysis tools at same level
- Not buried inside `monitoring/`
- Clear in documentation structure

✅ **Integration Points Clear**
```
costs/integration.py imports from:
  - tests/integration/framework-config.json
  - tests/perf/results/*/results.jtl
  - costs/fixtures/cost-config.json

Modified by:
  - scripts/run-benchmarks.py (calls CostSimulator after each framework)

Outputs to:
  - tests/perf/results/{framework}/cost-analysis.json
  - tests/perf/results/cost-comparison.json
```

## Integration Points

### 1. **With `scripts/run-benchmarks.py`**

```python
# In scripts/run-benchmarks.py, after framework test completes:

from costs.integration import CostSimulator
from costs.config import CloudProvider

simulator = CostSimulator()

# For each framework result:
cost_result = simulator.analyze_benchmark_result(
    jmeter_result_file=jmeter_path,
    framework_config=framework_registry[framework],
    profile=LoadProfile.SMALL,
    providers=[CloudProvider.AWS, CloudProvider.GCP, CloudProvider.AZURE]
)

# Save to tests/perf/results/{framework}/cost-analysis.json
cost_result.to_json(results_path / "cost-analysis.json")

# Update benchmark results
benchmark_results[framework]["cost"] = {
    "monthly_usd": cost_result.cost_aws.total,
    "efficiency_score": cost_result.efficiency.efficiency_score,
    "ranking": cost_result.efficiency.ranking
}
```

**Current workflow**:
```
1. Run framework tests
2. Run JMeter benchmarks
```

## Alternative: Why NOT Option 2 or 3

### Why NOT `monitoring/costs/`?

**Cons**:
- Cost analysis ≠ monitoring (separate concern)
- Buries it inside monitoring directory
- Harder to discover and understand structure
- Monitoring and cost analysis are independent (cost analysis doesn't need Prometheus)
- Monorepo pattern suggests peer directories

**When this makes sense**:
- If cost analysis is heavily integrated with monitoring (shared dashboards, shared metrics)
- Currently false - they're independent layers

### Why NOT `analysis/costs/`?

**Cons**:
- Over-engineering (only one thing in `analysis/` now)
- Extra nesting level
- Harder to find (two levels deep)
- No other analysis modules to group with

**When this makes sense**:
- If you plan to add `analysis/performance/`, `analysis/regression/`, etc. soon
- Consider waiting until you have 3+ analysis tools

## Recommendation Summary

| Aspect | Option 1 (`costs/`) | Option 2 (`monitoring/costs/`) | Option 3 (`analysis/costs/`) |
|--------|-------|-------|-------|
| **Clarity** | ⭐⭐⭐ Clear peer module | ⭐⭐ Mixed signals | ⭐⭐ Over-nested |
| **Scalability** | ⭐⭐⭐ Room to grow | ⭐⭐ Tied to monitoring | ⭐⭐⭐ Future-proof |
| **Discoverability** | ⭐⭐⭐ Top-level visibility | ⭐⭐ Buried | ⭐⭐ Extra nesting |
| **Independence** | ⭐⭐⭐ Standalone module | ⭐⭐ Tied to monitoring | ⭐⭐⭐ Independent |
| **Monorepo alignment** | ⭐⭐⭐ Matches pattern | ⭐⭐ Breaks pattern | ⭐⭐ Forces pattern |

**Winner: Option 1 (`costs/` subdirectory)**
