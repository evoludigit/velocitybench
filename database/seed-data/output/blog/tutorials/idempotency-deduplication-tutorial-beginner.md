# **Idempotency & Deduplication: Handling Retries Without Duplicate Work**

When you submit an order for pizza, place a payment, or book a flight, you expect the action to happen **once**—even if your internet flickers. But what if your request fails mid-transit and your client tries again? Without careful handling, you could end up with duplicate orders, overcharged accounts, or duplicate travel bookings—just like when someone forgets to remove a sticky note from the fridge.

This is where **idempotency** and **deduplication** come in. They ensure that retries don’t cause unintended side effects, even under the worst network conditions. In this post, we’ll explore:
- **Why retries cause problems** (network failures, client-side retries, and race conditions)
- **How idempotency makes retries safe** (with real-world examples in APIs and databases)
- **When deduplication steps in when idempotency isn’t possible**
- **Code examples in SQL, PostgreSQL, and API design**

## **The Problem: Why Retries Break Things**

Consider this common flow:

1. A client sends a `POST /orders` request.
2. The server receives it, creates an order, and returns a `201 Created` response.
3. Suddenly, the internet drops, and the client doesn’t receive the response.
4. The client **retries automatically** (due to exponential backoff or a library like `retry-request`).

If your API doesn’t handle retries gracefully, **the same order might get created twice**—or worse. Here’s why:

| Scenario               | Result Without Idempotency/Deduplication |
|------------------------|------------------------------------------|
| Duplicate orders       | Customer charged twice                   |
| Duplicate payments     | Bank transaction failure                 |
| Duplicate flight books | Two seats reserved for one person        |
| Race conditions        | Database inconsistencies (e.g., `COUNT` mismatches) |

---
## **The Solution: Idempotency + Deduplication**

### **1. Idempotency: Safe to Repeat**
An **idempotent operation** is one where:
- **Multiple identical requests** produce the **same result** as one request.
- **No side effects** occur on retries (e.g., no duplicates, no double charges).

#### **Idempotency in APIs**
APIs often use **idempotency keys**—unique identifiers generated by the client (e.g., a UUID) to track retries.

**Example: Idempotent Order Creation**
```http
POST /orders HTTP/1.1
Idempotency-Key: abc123-xyz456
Content-Type: application/json

{
  "product": "Pizza",
  "quantity": 1
}
```

**Server logic (Pseudocode):**
```python
def create_order(request):
    idempotency_key = request.headers.get("Idempotency-Key")

    if idempotency_key in idempotency_keys_seen:
        return "Already processed", 200  # For idempotent clients

    # Save order
    order = save_order(request.data)

    # Store idempotency key
    idempotency_keys_seen[idempotency_key] = True
    return order, 201
```

#### **Idempotency in Databases**
In SQL, idempotency is often enforced via **unique constraints** or **`ON CONFLICT` clauses** (PostgreSQL):

```sql
-- PostgreSQL: Insert or ignore duplicates
INSERT INTO orders (id, product, quantity)
VALUES ('abc123', 'Pizza', 1)
ON CONFLICT (id) DO NOTHING;

-- Or upsert (update if exists)
INSERT INTO orders (id, product, quantity)
VALUES ('abc123', 'Pizza', 1)
ON CONFLICT (id) DO UPDATE
SET product = EXCLUDED.product, quantity = EXCLUDED.quantity;
```

### **2. Deduplication: When Idempotency Isn’t Possible**
Some operations **can’t be made idempotent** (e.g., sending an email, triggering a workflow). Here, we **detect and prevent duplicates** instead.

**Example: Deduplicating Emails**
```sql
-- Use a unique index + error handling
INSERT INTO sent_emails (email_id, recipient, status)
VALUES ('123', 'user@example.com', 'pending')
ON CONFLICT (email_id) DO UPDATE
SET status = 'duplicate';

-- Or use a trigger to mark duplicates
CREATE TRIGGER check_for_duplicates
BEFORE INSERT ON sent_emails
FOR EACH ROW
WHEN (SELECT COUNT(*) FROM sent_emails WHERE email_id = NEW.email_id) > 1
BEGIN
    SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'Duplicate email';
END;
```

## **Implementation Guide**

### **Step 1: Choose an Idempotency Strategy**
| Strategy               | Pros                          | Cons                          |
|------------------------|-------------------------------|-------------------------------|
| **Idempotency Key**    | Simple, client-controlled     | Requires client cooperation   |
| **Database Checks**    | Automatic, no client changes  | Slower (extra queries)         |
| **API Versioning**     | Works for stateless APIs      | Not all APIs support retries  |

### **Step 2: Handle Network Failures Gracefully**
- **Exponential backoff**: Clients retry with increasing delays.
- **Idempotency headers**: Always include `Idempotency-Key` in retries.
- **Server-side logging**: Track failed requests for debugging.

### **Step 3: Database Optimizations**
- **Use transactions**: Ensure atomicity (e.g., `BEGIN; INSERT; COMMIT`).
- **Index on unique fields**: Speed up deduplication checks.
- **Eventual consistency**: For distributed systems, use queues (Kafka, RabbitMQ).

---

## **Common Mistakes to Avoid**

❌ **Assuming retries are rare** → Network failures happen; always plan for them.

❌ **Not validating idempotency keys** → Stored keys can leak sensitive data.

❌ **Ignoring race conditions** → Without locks, two clients might process the same request.

❌ **Over-relying on client-side idempotency** → Some clients (e.g., mobile apps) may not implement it correctly.

❌ **Not setting timeouts** → Long-running retries can exhaust resources.

---

## **Key Takeaways**
✅ **Idempotency** = Safe to retry (same input → same output).
✅ **Deduplication** = Prevents duplicates when idempotency isn’t possible.
✅ **Idempotency keys** = Best for APIs (client-managed).
✅ **Database constraints** = Best for internal systems (server-managed).
✅ **Always test retries** → Simulate network failures in integration tests.
