# **Schema Artifacts Generation: Automating Your Database & API Blueprints**

*How to Keep Your GraphQL/REST Schemas, Validation Reports, and Execution Plans in Sync—Automatically*

---

## **Introduction**

Imagine this scenario: You’re debugging a production incident where a query is failing with a cryptic error. You dig into the logs, only to realize the issue stems from a mismatch between your *GraphQL SDL* (Schema Definition Language) and your *execution plan*. Meanwhile, your team is maintaining separate files for the schema, validation rules, and database migrations, leading to confusion about what’s "real" and what’s outdated.

This is a common headache in backend development. When your API or database schema changes, you often end up with **three distinct artifacts**:
1. **The compiled schema** (how queries/execution plans are executed).
2. **The GraphQL SDL** (the human-readable definition for clients).
3. **Validation reports** (errors/warnings for developers).

But how do you ensure these artifacts stay in sync? How do you generate, version, and reuse them efficiently?

This is where the **Schema Artifacts Generation** pattern comes in—an approach to **automate the creation, versioning, and reuse of these artifacts** so your team never has to manually sync them again.

In this post, we’ll:
- Explore why keeping these artifacts in sync is painful (and how it causes bugs).
- Show how to **automate artifact generation** for GraphQL, SQL, and validation rules.
- Walk through **practical examples** in Go, Python, and JavaScript.
- Share implementation best practices and common pitfalls.

By the end, you’ll have a battle-tested way to keep your schema ecosystem clean, versioned, and debuggable.

## **The Problem: Schema Drift and Debugging Nightmares**

Schema artifacts shouldn’t exist in isolation. When they don’t sync, you get:

### **1. Mismatches Between SDL and Execution Plan**
A developer defines a GraphQL type in `schema.graphql`, but the compiled schema (e.g., in GraphQL-JS or Apollo) generates a different structure due to code-generated fields or mutations.

**Example:**
```graphql
# schema.graphql (intended)
type User @entity {
  id: ID!
  name: String!
}

# Compiled schema (actual execution plan)
type User {
  id: ID!
  name: String!
  _version: String!  # <-- Auto-generated by GraphQL Code Generator
}
```
Now, a frontend query like this:
```graphql
query { user { name } }
```
Fails silently because `_version` exists in the execution plan but isn’t in the SDL.

### **2. Outdated Validation Rules**
Your migration files include SQL schema validation, but your ORM (like Sequelize or TypeORM) generates its own DDL. A new `NOT NULL` constraint in the DB doesn’t match the ORM’s schema, causing runtime crashes.

```sql
-- Migration file (intended)
CREATE TABLE users (
  id INT PRIMARY KEY,
  name VARCHAR(255) NOT NULL,  -- <-- Missing in ORM's DDL
);
```

### **3. Debugging Hell**
When issues arise, you waste time hunting for the "source of truth":
- Is the issue in the SDL, the compiled schema, or the DB schema?
- Why did a recent PR break something?

Without artifacts, every change becomes a guessing game.

---

## **The Solution: Automated Schema Artifacts Generation**

The **Schema Artifacts Generation** pattern solves this by:

1. **Centralizing artifact generation** (e.g., SDL, SQL DDL, validation rules) in one place.
2. **Versioning artifacts** with your codebase (Git).
3. **Reusing artifacts** across services (e.g., GraphQL servers, DB migrations).
4. **Generating validation reports** to catch mismatches early.

### **Core Components**
| Component          | Purpose                                                                 |
|--------------------|-------------------------------------------------------------------------|
| **Artifact Generator** | Scripts/tools that create SDL, SQL, or validation rules from a "source of truth." |
| **Source of Truth** | A single definition (e.g., OpenAPI, GraphQL Codegen config, SQL migrations). |
| **Artifact Registry** | A directory (e.g., `./schema-artifacts`) storing generated files.       |
| **Validation Reports** | Linting tools that compare artifacts and flag discrepancies.           |

## **Implementation Guide**

We’ll demonstrate this pattern for **GraphQL** and **SQL** artifacts. The concepts apply to REST APIs (OpenAPI/Swagger) too.

### **Example 1: GraphQL SDL Generation (Go + GraphQL Codegen)**

#### **Problem**
You use `graphql-go` but manually edit `schema.graphql`. When the generated SDL changes (e.g., due to resolver logic), the file gets stale.

#### **Solution**
Use **GraphQL Codegen** to automate SDL generation from Go types.

##### **Step 1: Define Your Schema in Go**
```go
package models
import "github.com/99designs/gqlgen/graphql"

// User represents a schema type.
type User struct {
	ID   int
	Name string
}

// Query defines GraphQL queries.
type Query struct{}
func (*Query) Users() ([]*User, error) {
	return []*User{{ID: 1, Name: "Alice"}}, nil
}
```

##### **Step 2: Configure Codegen**
Install [`graphql-codegen`](https://github.com/99designs/graphql-go/tree/master/graphqlgen):
```bash
npm install -g @graphql-codegen/cli
```

Create a `.graphqlrc.yml`:
```yaml
schema: ./graphql/schema.graphql
generates:
  ./generated/graphql/schema.graphql:
    plugins:
      - schema-ast
```

##### **Step 3: Generate the SDL**
Run:
```bash
npx graphql-codegen
```
This updates `./generated/graphql/schema.graphql` automatically.

##### **Step 4: Sync with Git**
Add `./generated/graphql/schema.graphql` to `.gitignore` (it’s auto-generated), but commit the `.graphqlrc.yml` configuration.

##### **Step 5: Validate Consistency**
Use [`graphql-scalars`](https://github.com/99designs/gqlgen/tree/master/graphqlgen) or custom scripts to compare:
- The generated SDL (`./generated/schema.graphql`).
- The actual execution plan (from `gqlgen`'s introspection).

### **Example 2: SQL DDL Generation (Python + SQLAlchemy)**

#### **Problem**
You write raw SQL migrations, but SQLAlchemy generates its own DDL, causing drift.

#### **Solution**
Use SQLAlchemy’s **auto-reflection** to generate DDL from models, then validate against migrations.

##### **Step 1: Define Models**
```python
from sqlalchemy import Column, Integer, String
from sqlalchemy.ext.declarative import declarative_base

Base = declarative_base()

class User(Base):
    __tablename__ = "users"
    id = Column(Integer, primary_key=True)
    name = Column(String, nullable=False)  # <-- Validation rule
```

##### **Step 2: Generate DDL**
Add a `generate_ddl.py` script:
```python
from sqlalchemy import inspect

def generate_ddl_from_models(models):
    ddl = ""
    for model in models:
        inspector = inspect(model)
        ddl += f"CREATE TABLE {inspector.table_name} (\n"
        for col in inspector.columns:
            ddl += f"  {col.name} {col.type} {'NOT NULL' if col.nullable is False else ''},\n"
        ddl += ");\n"
    return ddl

if __name__ == "__main__":
    print(generate_ddl_from_models([User]))
```

Run it:
```bash
python generate_ddl.py > ./schema-artifacts/users.ddl.sql
```

##### **Step 3: Validate Against Migrations**
Compare the generated DDL with your migration files:
```python
import difflib

def validate_ddl(ddl, migration_sql):
    return difflib.unified_diff(
        ddl.splitlines(),
        migration_sql.splitlines(),
        fromfile="generated_ddl.sql",
        tofile="migration.sql"
    )

# Usage:
with open("./schema-artifacts/users.ddl.sql") as f:
    generated = f.read()
with open("./migrations/001_create_users.sql") as f:
    migration = f.read()
errors = validate_ddl(generated, migration)
if errors:
    raise ValueError("DDL mismatch!")
```

##### **Step 4: Automate in CI**
Add a GitHub Action to run validation on PRs:
```yaml
- name: Validate DDL
  run: python generate_ddl.py && python validate_ddl.py
```

### **Example 3: REST API Artifacts (OpenAPI + FastAPI)**

#### **Problem**
You define your API in FastAPI routes but lose track of the OpenAPI spec.

#### **Solution**
Generate OpenAPI specs from route definitions and validate them.

##### **Step 1: Define Routes**
```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()
class User(BaseModel):
    name: str

@app.post("/users")
def create_user(user: User):
    return {"id": 1, **user.dict()}
```

##### **Step 2: Generate OpenAPI**
Enable auto-generated OpenAPI:
```python
app.openapi_schema = {
    "openapi": "3.0.0",
    "info": {"title": "My API", "version": "1.0"},
    "paths": {"/users": {"post": {"summary": "Create user"}}},
}
```

##### **Step 3: Save to Artifacts**
Use `fastapi.openapi.json()` to dump the spec:
```bash
fastapi run --generate-openapi > schema-artifacts/api.openapi.json
```

##### **Step 4: Validate**
Use [`openapi-spec-validator`](https://github.com/OAI/OpenAPI-Specification) to check the spec:
```bash
npm install -g @apidevtools/swagger-parser
swagger-parser validate schema-artifacts/api.openapi.json
```

## **Common Mistakes to Avoid**

1. **Ignoring Versioning**
   - Don’t commit generated artifacts directly. Instead, version the *generator scripts* (e.g., `.graphqlrc.yml`, `generate_ddl.py`).

2. **Over-Reliance on Manual Edits**
   - If you edit `schema.graphql` manually, you’ll lose sync with Go/Python/SQL types. Always derive artifacts from "source of truth."

3. **Skipping Validation Reports**
   - Without diffing artifacts, you’ll never catch drift. Automate it in CI/CD.

4. **Tight Coupling to One Tool**
   - If you use GraphQL Codegen, ensure it works with your ORM (e.g., TypeORM) for SQL artifacts.

5. **Not Documenting the Pattern**
   - Add a `README` in your `schema-artifacts` directory explaining how to regenerate files.

---

## **Key Takeaways**

✅ **Automate artifact generation** from a single source of truth (e.g., Go types, SQLAlchemy models).
✅ **Version the generator, not the artifacts** (commit configs, not `.graphql` files).
✅ **Validate artifacts in CI/CD** to catch mismatches early.
✅ **Reuse artifacts** across GraphQL, SQL, and REST APIs.
✅ **Document the pattern** so new devs know how to regenerate everything.
