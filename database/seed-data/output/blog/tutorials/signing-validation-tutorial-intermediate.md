# **Signing Validation: Securing Your APIs Against Tampering**

*How to verify message integrity and authenticity in distributed systems*

## **Introduction**

As backend engineers, we increasingly deal with APIs that exchange sensitive data—payment details, user credentials, or internal system commands. Have you ever wondered how endpoints know whether the data they receive hasn’t been altered in transit?

This is where **signing validation** comes into play. It’s a security pattern that ensures data integrity by verifying whether messages (e.g., API requests, API responses, or internal service calls) were signed by an authorized party and haven’t been tampered with.

Unlike encryption (which hides content), signing validation focuses on **authenticity and integrity**: confirming the sender’s identity and ensuring the message hasn’t changed between creation and receipt.

But why is this critical? Without it, an attacker could intercept and alter a request (e.g., changing a `payment.amount` from `100` to `1000`). Signing validation prevents this by cryptographically binding a message to a known key.

In this guide, we’ll explore:
- **The real-world risks** of not validating signatures
- **How signing validation solves these problems**
- **Common signing patterns** (HMAC, JWT, asymmetric signatures)
- **Practical code examples** in Node.js and Python
- **Tradeoffs** and when to use each approach

Let’s dive in.

---

## **The Problem: Security Risks Without Signing Validation**

Imagine you’re building a payment processing system where merchants send requests to your API to charge customers. If an attacker intercepts and modifies a request—say, inflating the amount from `$10.00` to `$1,000.00`—what recourse does your system have?

**Without signing validation**, there’s no way to prove the request was legitimate. Attackers can exploit this to steal money, escalate privileges, or inject malicious data.

### **Common Attack Scenarios**
| Scenario | Impact | Example |
|----------|--------|---------|
| **Request Tampering** | Fraudulent transactions, data breaches | Changing `payment.productId` from `123` to `456` (which costs more) |
| **Replay Attacks** | Duplicate actions | Resubmitting a `withdrawal` request to drain an account |
| **Man-in-the-Middle (MITM) Attacks** | Hijacking authenticated sessions | Altering a JWT token’s `exp` claim to bypass time-based limits |

### **Real-World Example: The 2013 Yahoo Hack**
Early in 2013, Yahoo was breached due to weak session validation. The attacker exploited the fact that session cookies weren’t **signed with a secret key**, allowing them to forge valid session tokens.
*(Source: [Yahoo’s 2013 Security Breach](https://en.wikipedia.org/wiki/2013_Yahoo_security_breach))*

**Lesson:** Always validate signatures to detect tampered requests.

## **The Solution: Signing Validation Patterns**

Signing ensures two key properties:
1. **Integrity:** The message hasn’t been altered after signing.
2. **Authenticity:** The message was generated by a trusted source.

There are **three primary ways** to implement signing validation:
1. **HMAC (Hash-based Message Authentication Code)**
   - Symmetric signing (same key for signing and validation).
   - Fast, but requires secure key sharing.
2. **Asymmetric Signing (RSA, ECDSA)**
   - Uses public/private key pairs (private key signs, public key validates).
   - More secure for public APIs but slower.
3. **JWT (JSON Web Tokens)**
   - Combines signing with a structured payload for tokens.
   - Common for stateless authentication but requires careful HMAC/RSA use.

Let’s explore each in detail.

---

## **Components/Solutions**

### **1. HMAC Signing (Symmetric)**
HMAC uses a shared secret key to sign messages. Both the client and server must know this key.

#### **When to Use**
- Internal services where keys can be securely distributed.
- Low-latency applications (HMAC is faster than RSA).

#### **Example Workflow**
1. Client computes `hmacSha256(secretKey, requestData)`.
2. Client sends `requestData` + `signature` to the server.
3. Server recomputes HMAC and compares it to the received signature.

---

### **2. Asymmetric Signing (RSA/ECDSA)**
Asymmetric signing uses a public/private key pair. The private key signs the message; the public key validates it.

#### **When to Use**
- Public APIs where clients don’t share a secret key.
- High-security scenarios (e.g., blockchain, financial systems).

#### **Example Workflow**
1. Server generates a public/private key pair.
2. Client requests the public key (e.g., via `/v1/keys`).
3. Client signs the message with their private key (or the server’s public key, depending on use case).
4. Server validates using its private key (or client’s public key).

---

### **3. JWT with Signing**
JWTs include a signed payload, often using HMAC or RSA.

#### **When to Use**
- Stateless authentication (e.g., OAuth, API tokens).
- When you need a compact, self-contained token.

#### **Example Workflow**
1. Client receives a JWT with:
   ```
   {
     "sub": "user123",
     "exp": 1735689600
   }
   ```
   Signed with HMAC or RSA.
2. Server verifies the signature before processing the request.

## **Code Examples**

### **1. HMAC Signing in Node.js**
Let’s simulate a merchant sending a payment request to your API.

#### **Server-Side (Validation)**
```javascript
// server.js
const crypto = require('crypto');
const SECRET_KEY = 'your-256-bit-secret-key-here'; // Should be in env vars!

// Simulate a request from a merchant
const requestData = {
  merchantId: 'merchant_123',
  amount: 100,
  productId: 'gift_card_abc',
  timestamp: Date.now(),
};

// Client sends: { data, signature }
const clientSignature = 'e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855'; // Precomputed elsewhere

// Validate the signature
const hmac = crypto.createHmac('sha256', SECRET_KEY);
const computedSignature = hmac.update(JSON.stringify(requestData)).digest('hex');

if (computedSignature !== clientSignature) {
  console.error('Signature validation failed!');
  return { error: 'Invalid signature' };
}

console.log('Signature valid! Processing payment...');
```

#### **Client-Side (Signing)**
```javascript
// client.js
const crypto = require('crypto');
const requestData = {
  merchantId: 'merchant_123',
  amount: 100,
  productId: 'gift_card_abc',
  timestamp: Date.now(),
};

const SECRET_KEY = 'your-256-bit-secret-key-here';
const hmac = crypto.createHmac('sha256', SECRET_KEY);
const signature = hmac.update(JSON.stringify(requestData)).digest('hex');

console.log('Sending request with signature:', signature);
```

### **2. Asymmetric Signing in Python (RSA)**
Now, let’s assume your API requires clients to sign requests with their own private key (e.g., for high-security scenarios).

#### **Server-Side (Validation)**
```python
# server.py
import json
from cryptography.hazmat.primitives import serialization, hashes
from cryptography.hazmat.primitives.asymmetric import padding
from cryptography.hazmat.backends import default_backend

# Load the public key (expected from the client)
with open('client_public_key.pem', 'rb') as key_file:
    public_key = serialization.load_pem_public_key(
        key_file.read(),
        backend=default_backend()
    )

# Simulate a request
request_data = {
    "merchantId": "merchant_123",
    "amount": 100,
    "productId": "gift_card_abc",
    "timestamp": "2024-01-01T12:00:00Z"
}

# Client sends: { data, signature }
client_signature = b'\x12\x34...'  # Hex-encoded signature from client

try:
    # Verify the signature
    public_key.verify(
        client_signature,
        json.dumps(request_data, separators=(',', ':')).encode(),
        padding.PSS(
            mgf=padding.MGF1(hashes.SHA256()),
            salt_length=padding.PSS.MAX_LENGTH
        ),
        hashes.SHA256()
    )
    print("Signature valid! Processing payment.")
except Exception as e:
    print("Signature validation failed:", e)
```

#### **Client-Side (Signing)**
```python
# client.py
import json
from cryptography.hazmat.primitives import serialization, hashes
from cryptography.hazmat.primitives.asymmetric import padding
from cryptography.hazmat.backends import default_backend

# Load private key (client's secret key)
with open('client_private_key.pem', 'rb') as key_file:
    private_key = serialization.load_pem_private_key(
        key_file.read(),
        password=None,
        backend=default_backend()
    )

request_data = {
    "merchantId": "merchant_123",
    "amount": 100,
    "productId": "gift_card_abc",
    "timestamp": "2024-01-01T12:00:00Z"
}

# Sign the request
signature = private_key.sign(
    json.dumps(request_data, separators=(',', ':')).encode(),
    padding.PSS(
        mgf=padding.MGF1(hashes.SHA256()),
        salt_length=padding.PSS.MAX_LENGTH
    ),
    hashes.SHA256()
)

print("Sending signature:", signature.hex())
```

### **3. JWT Signing Validation (Node.js)**
For stateless APIs, JWTs are a common choice. Let’s validate a JWT with HMAC.

#### **Server-Side Validation**
```javascript
// server.js
const jwt = require('jsonwebtoken');

const SECRET_KEY = 'your-jwt-secret-key-here';

const token = 'eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...'; // Example JWT

  console.log('Valid token!', decoded);
} catch (err) {
  console.error('Invalid token:', err.message);
}

## **Implementation Guide**

### **Step 1: Choose Your Signing Method**
| Method       | Use Case                          | Pros                          | Cons                          |
|--------------|-----------------------------------|-------------------------------|-------------------------------|
| **HMAC**     | Internal services, high throughput | Fast, simple key management   | Key must be shared securely   |
| **RSA**      | Public APIs, high security       | No shared secrets             | Slower, key management complex |
| **JWT**      | Stateless auth, tokens           | Compact, self-contained       | Requires careful key rotation |

### **Step 2: Generate and Distribute Keys**
- **HMAC:** Store the secret in environment variables (e.g., `process.env.SIGNING_KEY`).
- **RSA:** Use tools like OpenSSL to generate keys:
  ```bash
  openssl genpkey -algorithm RSA -out private_key.pem -pkeyopt rsa_keygen_bits:2048
  openssl rsa -pubout -in private_key.pem -out public_key.pem
  ```
- **JWT:** Use HTTPS to protect the `/.well-known/jwks.json` endpoint if using public keys.

### **Step 3: Implement Signature Validation**
- For **HMAC**, compute the signature on the server and compare it to the client’s signature.
- For **RSA**, verify the signature using the public key.
- For **JWT**, ensure `algorithm` is hardcoded (e.g., `HS256` or `RS256`).

### **Step 4: Handle Errors Gracefully**
- Reject requests with invalid signatures immediately.
- Log signature failures (without exposing sensitive data).
- Consider rate-limiting to prevent brute-force attacks.

### **Step 5: Rotate Keys Periodically**
- HMAC: Rotate the secret key during deployment.
- RSA: Use short-lived keys (e.g., rotate every 30 days).
- JWT: Implement key rotation in `jwks_uri`.

## **Common Mistakes to Avoid**

### **1. Not Using HTTPS**
Signing alone doesn’t protect against MITM attacks. Always use HTTPS to encrypt traffic before signing.

### **2. Hardcoding Secrets**
Never hardcode keys in your code. Use environment variables or a secrets manager (e.g., AWS Secrets Manager, HashiCorp Vault).

### **Example of a Security Risk**
```javascript
// ❌ BAD: Hardcoded key
const SECRET_KEY = 'supersecret'; // Anyone with access to this file knows it!
```

### **3. Ignoring Timestamp Validation**
Always validate timestamps (e.g., `timestamp` in HMAC examples) to prevent replay attacks.

### **4. Over-Reliance on JWT Without Short Expiry**
JWTs with long expiry times (e.g., 1 year) create security risks. Use short-lived tokens + refresh tokens.

### **5. Not Testing Edge Cases**
- Test with malformed data (e.g., invalid JSON).
- Test with tampered signatures.
- Test key rotation scenarios.
