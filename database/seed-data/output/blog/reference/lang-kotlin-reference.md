**#[Pattern] Kotlin Language Patterns – Reference Guide**

---

### **Overview**
The **Kotlin Language Patterns** reference guide provides structured insights into idiomatic Kotlin constructs, emphasizing best practices, performance optimizations, and common anti-patterns. Whether you’re designing object-oriented (OO) or functional paradigms, this guide ensures clarity and maintainability. Topics include **null safety**, **extension functions**, **coroutines**, **smart casts**, **destructuring declarations**, and **data classes**, along with code samples, trade-offs, and migration tips. Ideal for developers seeking to leverage Kotlin’s expressive syntax while avoiding pitfalls.

### **1. Schema Reference (Key Concepts & Best Practices)**

| **Category**               | **Pattern/Concept**               | **Description**                                                                                                                                                                                                                                                                                          | **Example**                                                                                                                                                                                                                                                                                     | **Best Practices**                                                                                                                                                                                                                                                                                                                                 | **Anti-Patterns**                                                                                                                                                                                                                                                                                  |
|----------------------------|-----------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Null Safety**            | `Nullable vs Non-null Types`      | Differentiates between `String?` (nullable) and `String` (non-null) to enforce safe API designs and reduce NPEs.                                                                                                                                                                                                     | ```kotlin val name: String? = null // Nullable val age: Int = 25 // Non-null ```                                                                                                                                                                                                                 | - Prefer non-null types and use `!!` only in tested contexts.<br>- Use default values (`?:`) for nullable inputs.<br>- Leverage `lateinit` for properties initialized after initialization.<br>- Tools: `kotlinx.coroutines.runCatching`.                                                                                                                                         | - Ignoring nullability warnings.<br>- Overusing `let`/`apply` on nullable objects without safe-call chains (`?.`/`!!`).                                                                                                                                                   |
| **Extension Functions**    | `Custom Null-Safe Methods`        | Extends existing classes (e.g., `String`) without inheritance, enabling reusable utility functions.                                                                                                                                                                                                         | ```kotlin fun String.ifEmptyOrNull(): String? { return if (this.isEmpty()) null else this } ```                                                                                                                                                                                                                   | - Group extensions by semantic purpose (e.g., `StringExtensions`, `CollectionExtensions`).<br>- Document performance implications (e.g., extension functions may incur reflection overhead).<br>- Avoid shadowing built-in functions or library extensions.                                                                                                 | - Overloading extension functions with ambiguous signatures.<br>- Extending sealed classes or interfaces (potential future compatibility issues).                                                                                                                                     |
| **Coroutines**             | `Suspend Functions`               | Asynchronous programming with lightweight threads via `suspend` keywords.                                                                                                                                                                                                                                      | ```kotlin suspend fun fetchData(): String { delay(1000) return "Data" } ```                                                                                                                                                                                                                                           | - Structure coroutines hierarchically (e.g., `viewModelScope` for UI-related tasks).<br>- Use `Dispatchers.IO` for blocking I/O.<br>- Handle cancellation via `Job` or `CoroutineScope.cancel()`.<br>- Prefer `runBlocking` only in tests/main threads.                                                                                                                                 | - Blocking coroutines (e.g., using `suspend` in `Main` thread without `withContext(Dispatchers.Main)`).<br>- Exposing coroutines directly in APIs (wrap in `Flow`/`StateFlow`).                                                                                                 |
| **Smart Casts**            | `Type Checking + Casting`         | Eliminates redundant `as` casts when Kotlin deduces types after `is` checks.                                                                                                                                                                                                                                      | ```kotlin when (value) { is Int -> println(value + 1) // Smart cast ```                                                                                                                                                                                                                                           | - Combine with sealed classes for exhaustive type-checking.<br>- Prefer smart casts over `when (value is Type) {}` for readability.                                                                                                                                                                                                 | - Assuming smart casts work across interfaces (use `is` checks first).<br>- Overusing `as!` to avoid null checks (may crash).                                                                                                                                                  |
| **Destructuring Declarations** | `Data Class Decomposition`   | Extracts properties into variables via `to()` syntax for tuples-like unpacking.                                                                                                                                                                                                                            | ```kotlin val (x, y) = Point(1, 2) // x=1, y=2 ```                                                                                                                                                                                                                                                     | - Use for single-purpose patterns (e.g., coordinates, responses).<br>- Document structs with `data class` for auto-generated `copy()`, `equals()`, and `toString()`.                                                                                                                                                             | - Overusing for complex objects (use named parameters instead).<br>- Ignoring immutability (mutable data classes can lead to state leaks).                                                                                                                                               |
| **Data Classes**           | `Auto-Generated Boilerplate`      | Reduces manual implementation of `equals()`, `hashCode()`, `toString()`, and `copy()`.                                                                                                                                                                                                                     | ```kotlin data class User(val name: String, val age: Int) ```                                                                                                                                                                                                                                                 | - Favor `data class` over `class` for value objects.<br>- Extend with custom logic via `componentN()` for custom destructuring.<br>- Use `sealed class` for exhaustiveness checks.<br>- Leverage `copy()` for immutable updates.                                                                                                                                 | - Overusing for entire domain models (separate DTOs for APIs).<br>- Forgetting `override val component1` in custom destructuring.                                                                                                                                                 |
| **Sealed Classes**         | `Exhaustive Type-Safe Hierarchies** | Restricts subclasses to a predefined set, enforcing exhaustive `when` clauses.                                                                                                                                                                                                                          | ```kotlin sealed class Result { data class Success(val data: String) : Result() data class Error(val msg: String) : Result() } ```                                                                                                                                                                                | - Use with `sealed interface` (Kotlin 1.6+) for interfaces.<br>- Document all possible subclasses in `when` blocks.<br>- Combine with `sealed` for sealed interfaces in library design.                                                                                                                                         | - Incomplete `when` clauses (IDE warnings may not catch all cases).<br>- Subclassing without overriding `equals()`/`hashCode()`.                                                                                                                                                     |
| **Lazy Initialization**    | `Lazy Properties`                 | Defers initialization until first access via `lazy(Mode)`. Useful for expensive or thread-safe resources.                                                                                                                                                                                                          | ```kotlin val expensiveResource: String by lazy { loadResource() } ```                                                                                                                                                                                                                                               | - Use `lazy(LazyThreadSafetyMode.SYNCHRONIZED)` for shared resources.<br>- Initialize in companion object for static-like behavior.<br>- Avoid recursion in lazy properties (may deadlock).                                                                                                                              | - Overusing for all properties (adds synchronization overhead).<br>- Forgetting to handle `LazyThreadSafetyMode` in concurrent environments.                                                                                                                                   |
| **Delegation**             | `by Lazy`, `by Observable`        | Delegates property logic to other objects (e.g., `lazy`, `Observable` for MVVM).                                                                                                                                                                                                                             | ```kotlin val name: String by lazy { loadName() } ```                                                                                                                                                                                                                                                     | - Use `by Delegate` for observable properties (e.g., `androidx.lifecycle.Observable`).<br>- Prefer `by` for custom logic (e.g., `by mapOf` for immutable maps).                                                                                                                                                             | - Over-delegating (increases complexity).<br>- Ignoring thread-safety in observable properties.                                                                                                                                                                                        |
| **Inline Classes**         | `@JvmInline`                      | Zero-overhead abstraction for lightweight wrappers (e.g., `UUID`).                                                                                                                                                                                                                                         | ```kotlin @JvmInline value class UUID(val value: String) ```                                                                                                                                                                                                                                                     | - Use for performance-critical types (e.g., enums, small structs).<br>- Leverage `@JvmInline` to avoid boxed primitives.<br>- Document that inline classes are opaque (cannot be subclassed).                                                                                                                                                     | - Overusing for mutable state (inline classes are immutable by design).<br>- Mixing with nullable types (`UUID?`).                                                                                                                                                                |
| **Functional Patterns**   | `Higher-Order Functions`         | Functions that accept/take functions (e.g., `map`, `filter`).                                                                                                                                                                                                                                          | ```kotlin val evens = numbers.filter { it % 2 == 0 } ```                                                                                                                                                                                                                                                     | - Prefer `forEach` for side effects, `map`/`reduce` for transformations.<br>- Use `Sequence` for lazy collections.<br>- Document performance impacts (e.g., `map` vs. `forEach`).                                                                                                                                                 | - Nested lambdas (reduce readability).<br>- Overusing `apply`/`run` (favor explicit builders).                                                                                                                                                                                     |
| **smart casts**            | `Type-Safe Operations`            | Removes redundant casts after type checks (e.g., `if (x is Int) x++`).                                                                                                                                                                                                                                     | ```kotlin val obj: Any = 42 if (obj is Int) obj += 1 // Smart cast ```                                                                                                                                                                                                                                       | - Combine with `sealed` classes for exhaustive checks.<br>- Use `when (value)` with `is` checks for clarity.<br>- Document limitation: smart casts don’t work with `Any?`.                                                                                                                                                   | - Assuming smart casts work with `when` (exhaustiveness required).<br>- Overusing `as` before `is` (redundant).                                                                                                                                                                            |
| **Coroutine Scopes**       | `Lifecycle-Aware Scopes`          | Binds coroutines to lifecycle events (e.g., `viewLifecycleOwnerScope`).                                                                                                                                                                                                                                | ```kotlin lifecycleScope.launch { // UI-related work ```                                                                                                                                                                                                                                               | - Prefer `viewModelScope` for ViewModels, `lifecycleScope` for Activities/Fragments.<br>- Cancel scopes in `onDestroy()`.<br>- Use `CoroutineExceptionHandler` for error handling.                                                                                                                                                          | - Leaking scopes (e.g., `GlobalScope`).<br>- Mixing scopes (e.g., `viewModelScope` + `lifecycleScope`).                                                                                                                                                                         |
| **Type Aliases**           | `Complex Type Simplification`     | Replaces verbose types with aliases (e.g., `typealias PersonList = List<Person>`).                                                                                                                                                                                                                         | ```kotlin typealias Callback = (String) -> Unit ```                                                                                                                                                                                                                                                     | - Use for readability (e.g., generic types).<br>- Avoid for one-off usages.<br>- Document when aliases are generic.<br>- Prefer aliases over nested generic types (e.g., `List<List<String>>` → `typealias NestedList = List<List<String>>`).                                                                                         | - Overusing for simple types.<br>- Breaking API contracts by renaming aliases.                                                                                                                                                                                                   |

### **2. Query Examples (Code Samples)**

#### **Null Safety**
```kotlin
// Safe call chain
val name: String? = null
val length = name?.length ?: 0 // Returns 0 if null

// Elvis operator with default
val config = getConfig() ?: throw IllegalStateException("Config missing")
```

#### **Extension Functions**
```kotlin
fun String.truncate(maxLength: Int): String =
    if (length <= maxLength) this else substring(0, maxLength)

// Usage
"HelloWorld".truncate(5) // "Hello"
```

#### **Coroutines**
```kotlin
// Sequential calls
viewLifecycleOwner.lifecycleScope.launch {
    val data1 = withContext(Dispatchers.IO) { fetchData1() }
    val data2 = withContext(Dispatchers.IO) { fetchData2() }
    // Combine results
}

// Parallel calls
val deferred1 = async { fetchData1() }
val deferred2 = async { fetchData2() }
val combined = deferred1.await() + deferred2.await()
```

#### **Sealed Classes**
```kotlin
sealed class Weather {
    data class Sunny(val temperature: Int) : Weather()
    data class Rainy(val precipitation: String) : Weather()
}

fun handleWeather(weather: Weather) {
    when (weather) {
        is Weather.Sunny -> println("It's ${weather.temperature}°C")
        is Weather.Rainy -> println("Rain: ${weather.precipitation}")
    }
}
```

#### **Data Classes**
```kotlin
data class User(val name: String, val age: Int) {
    // Custom logic
    fun isAdult() = age >= 18
}

// Destructuring
val (name, age) = User("Alice", 25)
println("$name is ${if (age.isAdult()) "adult" else "minor"}")
```

#### **Lazy Initialization**
```kotlin
// Thread-safe lazy property
val expensiveResource: String by lazy(LazyThreadSafetyMode.SYNCHRONIZED) {
    loadResource() // Loads only once
}
```

#### **Inline Classes**
```kotlin
@JvmInline value class Meter(val value: Double) // Zero overhead

fun printDistance(distance: Meter) {
    println("${distance.value} m")
}
```

#### **Functional Patterns**
```kotlin
// Filter and map
val numbers = listOf(1, 2, 3, 4)
val evens = numbers.filter { it % 2 == 0 }.map { it * 2 } // [4, 8]

// Sequence for lazy evaluation
val largeList = Sequence { // Generates on demand
    yield(1)
    yield(2)
    // ...
}.map { it * 2 }
```

| **[Kotlin Coroutines Best Practices](#)** | Leverages structured concurrency with `Job`, `Cancellation`, and `Dispatchers`.                                                                                                                                                                                                                     | For async tasks (API calls, UI updates, background workers).                                                                                                                                                                                                                                            |
