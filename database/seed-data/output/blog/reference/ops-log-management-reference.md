# **[Pattern] Log Management Patterns – Reference Guide**

---

## **Overview**
Log Management Patterns provide structured approaches to collecting, processing, storing, analyzing, and acting on log data generated by applications, infrastructure, and systems. Effective log management ensures traceability, debugging efficiency, compliance, and operational insights. This guide outlines key patterns for centralized, scalable, and actionable log management, including **Structured Logging**, **Log Centralization**, **Log Enrichment**, **Log Retention & Archiving**, **Log Querying & Analytics**, and **Alerting & Response**. Implementing these patterns optimizes log lifecycle management while reducing operational overhead.

---

## **Implementation Details**
### **Key Concepts**
1. **Log Generation**:
   - Structured logs (JSON, key-value format) improve parsing and querying.
   - Example: `{ "timestamp": "2024-05-20T12:00:00Z", "level": "ERROR", "service": "auth", "user": "admin", "message": "Invalid credentials" }`

2. **Log Egress**:
   - Use agents (e.g., Filebeat, Fluentd) or SDKs (e.g., AWS SDK, OpenTelemetry) to ship logs to collectors.

3. **Log Processing**:
   - **Normalization**: Convert unstructured logs into a consistent format.
   - **Enrichment**: Add context (e.g., geo-location, user metadata) via lookup tables or APIs.
   - **Parsing**: Extract structured fields using regex or NLP (e.g., Splunk, ELK).

4. **Storage**:
   - **Hot Storage** (e.g., Elasticsearch): Fast retrieval for real-time analytics.
   - **Cold Storage** (e.g., S3, Snowflake): Cost-effective long-term retention.

5. **Querying & Analysis**:
   - Use tools like Elasticsearch (Kibana), Grafana, or Presto for ad-hoc or scheduled queries.
   - Define metrics (e.g., error rates, latency percentiles).

6. **Actionability**:
   - **Alerting**: Trigger alerts via Slack, PagerDuty, or webhooks (e.g., "ERROR count > 5 in 5 minutes").
   - **Response Automation**: Integrate with incident management tools (e.g., Jira, ServiceNow).

## **Schema Reference**
Below is a reference schema for structured log events. Customize fields based on your application’s needs.

| **Field**            | **Type**       | **Description**                                                                 | **Example Values**                          |
|----------------------|----------------|---------------------------------------------------------------------------------|---------------------------------------------|
| `timestamp`          | `ISO8601`      | When the log event occurred.                                                    | `2024-05-20T12:00:00.000Z`                |
| `level`              | `string`       | Severity of the event (DEBUG, INFO, WARN, ERROR, FATAL).                        | `ERROR`                                    |
| `source`             | `string`       | Originating service/component.                                                  | `auth-service`, `api-gateway`              |
| `service`            | `string`       | Application/service name.                                                       | `order-processing`                         |
| `user`               | `string`       | End-user identifier (if applicable).                                            | `user-12345`                               |
| `action`             | `string`       | User or system action (e.g., `login`, `payment_failed`).                       | `authentication_failed`                   |
| `status`             | `integer`      | HTTP status code or custom status (e.g., `403`, `500`).                        | `403`                                      |
| `message`            | `string`       | Human-readable description.                                                     | `Invalid API key`                          |
| `context`            | `object`       | Additional metadata (e.g., `request_id`, `ip_address`, `region`).               | `{"request_id": "abc123", "ip": "192.0.2.1"}`|
| `duration_ms`        | `float`        | Latency of the operation.                                                       | `150.2`                                    |
| `tags`               | `array`        | Categorical labels (e.g., `["db-query", "slow"]`).                             | `["payment", "timeout"]`                   |
| `correlation_id`     | `string`       | Unique identifier for tracing across services.                                  | `x-correlation-456`                        |

---
**Note**: Use a standardized schema (e.g., OpenTelemetry’s LogRecord) for interoperability.

## **Query Examples**
### **1. Error Rate Analysis**
**Tool**: Elasticsearch (Kibana)
**Query**:
```json
{
  "query": {
    "bool": {
      "must": [
        { "match": { "level": "ERROR" } },
        { "range": { "timestamp": { "gte": "now-1h", "lte": "now" } } }
      ]
    }
  },
  "aggs": {
    "errors_by_service": { "terms": { "field": "service.keyword" } },
    "error_count": { "value_count": { "field": "correlation_id" } }
  }
}
```
**Output**:
- Breakdown of errors by service (e.g., `auth-service: 42`, `payment: 8`).
- Total unique errors in the last hour.

---

### **2. User Login Failures**
**Tool**: Grafana (Prometheus/Elasticsearch)
**Query** (Lua/Elasticsearch DSL):
```lua
{
  "query": {
    "bool": {
      "must": [
        { "match": { "action": "login" } },
        { "match": { "level": "ERROR" } },
        { "range": { "timestamp": { "gte": "now-7d" } } }
      ]
    }
  }
}
```
**Visualization**:
- Time-series chart of failed logins per day.
- Compare with successful logins (`action: "login", level: "INFO"`).

---

### **3. Slow API Endpoints**
**Tool**: Presto/Spark SQL
**Query**:
```sql
SELECT
  service,
  action,
  COUNT(*) AS call_count,
  AVG(duration_ms) AS avg_latency,
  PERCENTILE_APPROX(duration_ms, 0.95) AS p95_latency
FROM logs
WHERE timestamp > NOW() - INTERVAL '1 day'
  AND duration_ms > 500
GROUP BY service, action
ORDER BY avg_latency DESC;
```
**Output**:
- Top 5 slow endpoints with latency percentiles (e.g., `p95_latency = 1.2s`).

---

### **4. Anomaly Detection (Alert)**
**Tool**: AWS CloudWatch Logs Insights
**Query**:
```sql
stats count(*) as TotalErrors by service
| filter level = "ERROR"
| where $Timestamp > ago(1h)
| sort TotalErrors desc
| limit 10
```
**Alert Condition**:
- If `TotalErrors > 10` for any service, trigger a Slack alert:
  ```
  🚨 *Error Spike Detected* 🚨
  Service: [service]
  Errors: [TotalErrors] (threshold: 10)
  ```

## **Log Enrichment Patterns**
| **Pattern**               | **Description**                                                                 | **Implementation Example**                          |
|---------------------------|---------------------------------------------------------------------------------|-----------------------------------------------------|
| **Lookup Tables**         | Add static metadata (e.g., user roles, IP geolocation).                        | Elasticsearch ingest pipeline: `add_field: { "user": { "role": "predefined_roles" } }` |
| **API Integration**       | Fetch real-time data (e.g., user details from a database).                      | Fluentd plugin: `@in_elasticsearch { host: "es" }; filter { enrich { user_api { url: "https://userdb/api/user/%{user}" } } }` |
| **Correlation**           | Link logs to traces/spans (e.g., OpenTelemetry’s `trace_id`).                  | Add `trace_id` to logs: `log_event.trace_id = trace.get_id()` |

## **Log Retention & Archiving Patterns**
| **Pattern**               | **Use Case**                                  | **Implementation**                                  |
|---------------------------|-----------------------------------------------|-----------------------------------------------------|
| **Tiered Storage**        | Hot logs (Elasticsearch) → Cold (S3/HDFS).   | Elasticsearch ILM (Index Lifecycle Management).    |
| **Compression**           | Reduce storage costs for archived logs.       | Use `gzip` or `zstd` for S3 storage.               |
| **Purging Old Logs**      | Automate cleanup of logs older than X days.    | AWS Lambda triggered by S3 version events.         |
| **Immutable Logs**        | Ensure logs cannot be altered (compliance).   | Write to WORM (Write Once, Read Many) storage (e.g., AWS S3 Object Lock). |

## **Alerting & Response Patterns**
| **Pattern**               | **Description**                                                                 | **Example Tools**                                  |
|---------------------------|---------------------------------------------------------------------------------|---------------------------------------------------|
| **Threshold-Based**       | Alert when log count/metric exceeds a threshold.                               | Elastic Alerts, Prometheus Alertmanager.         |
| **Anomaly Detection**     | Use ML to detect deviations (e.g., sudden spike in 4xx errors).               | Datadog Anomaly Detection, AWS GuardDuty.         |
| **Context-Aware**         | Alert based on log context (e.g., "ERROR + user: admin").                     | Splunk Correlation Search.                       |
| **Automated Response**    | Trigger remediation (e.g., restart service).                                   | AWS Lambda + CloudWatch Events.                   |

**Example Alert Rule (Elasticsearch)**:
```json
{
  "condition": {
    "comparison": "gt",
    "aggregation": "count",
    "value": 10,
    "field": "level"
  },
  "filter": {
    "bool": { "must": [ { "term": { "level": "ERROR" } }, { "range": { "timestamp": { "gte": "now-5m" } } } ] }
  }
}
```

## **Security Considerations**
1. **Encryption**:
   - At rest: Use AES-256 (e.g., KMS, TLS for S3).
   - In transit: Enforce TLS for log shippers (e.g., Fluentd → Elasticsearch).

2. **Access Control**:
   - Role-based access (e.g., `log-reader`, `log-writer`) via IAM/PBAC.
   - Audit logs for log management operations.

3. **Data Minimization**:
   - Avoid logging PII (e.g., passwords) or sensitive metadata.
   - Use redaction tools (e.g., OpenTelemetry’s `logger.addAttribute()` with masking).

4. **Integrity**:
   - Use checksums (e.g., SHA-256) to detect log tampering.
   - Implement log signing (e.g., AWS Signer for S3 logs).
