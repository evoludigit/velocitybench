# **[Pattern] Logging Troubleshooting Reference Guide**

---

## **Overview**
Logging troubleshooting is a systematic approach to diagnosing, analyzing, and resolving issues in software applications by reviewing logs. This pattern provides structured methods for inspecting log data, interpreting errors, and addressing performance bottlenecks. It covers log sources (application, server, database), filtering techniques, and common troubleshooting workflows. Effective logging enables proactive issue detection, reduces downtime, and improves system reliability. This guide outlines key concepts, log schema structures, query examples, and related patterns to ensure comprehensive logging diagnostics.

---

## **Key Concepts**
### **1. Log Sources**
- **Application Logs**: Generated by business logic (e.g., errors, warnings, debug messages).
- **Server Logs**: Contain infrastructure details (e.g., web server, load balancer events).
- **Database Logs**: Track SQL queries, connection issues, and transaction failures.
- **Audit Logs**: Record user actions, security events, or configuration changes.
- **System Logs**: OS-level events (e.g., kernel errors, process crashes).

### **2. Log Levels**
Standardized severity levels (per [RFC 5424](https://tools.ietf.org/html/rfc5424)):
| Level   | Description                          | Use Case                                  |
|---------|--------------------------------------|-------------------------------------------|
| DEBUG   | Detailed diagnostic information.     | Development/debugging.                    |
| INFO    | Confirmation that sections ran.      | Normal operation flow.                   |
| WARN    | Potential issues (non-fatal).        | Anomalies needing attention.              |
| ERROR   | Serious problems affecting functionality. | Critical failures.                     |
| CRITICAL| Severe errors requiring immediate attention. | System-wide crashes.                     |

### **3. Log Structures**
Logs typically follow a standardized format:
```
TIMESTAMP LEVEL COMPONENT PROCESS_ID THREAD_ID MESSAGE [ADDITIONAL_DATA]
```
Example:
```
2024-05-15T14:30:45.123Z ERROR database-connector 12345 thread-1 "Failed to connect to DB: Timeout exceeded."
```

### **4. Common Log Patterns**
- **Error Stack Traces**: Include method calls leading to failures.
- **Performance Metrics**: Latency, throughput, or resource usage spikes.
- **Correlation IDs**: Unique identifiers to track requests across logs.

### **5. Tools & Technologies**
| Tool/Standard          | Purpose                                  | Example Use Cases                     |
|------------------------|------------------------------------------|---------------------------------------|
| **ELK Stack**          | Log aggregation, analysis, and visualization. | Real-time monitoring, alerting.       |
| **Fluentd/Fluent Bit** | Log collection and forwarding.          | Efficient log shipping to centralized storage. |
| **Prometheus + Grafana** | Metrics + logs correlation.               | Monitoring SLOs/SLA breaches.         |
| **Structured Logging** (JSON) | Machine-readable logs for parsing.      | Querying with tools like Splunk.      |
| **OpenTelemetry**      | Distributed tracing + logs.              | Tracing latency in microservices.     |

## **Schema Reference**
Below are standardized schemas for common log types (adaptable to JSON, CSV, or database tables):

### **1. Application Log Schema**
| Field               | Type         | Description                                                                 | Example Value                     |
|---------------------|--------------|-----------------------------------------------------------------------------|-----------------------------------|
| `timestamp`         | datetime     | When the log entry was generated.                                          | `2024-05-15T14:30:45.123Z`       |
| `level`             | string       | Severity level (DEBUG, INFO, WARN, ERROR, CRITICAL).                        | `ERROR`                           |
| `component`         | string       | Module/generated by (e.g., `auth-service`, `payment-gateway`).               | `user-auth`                       |
| `process_id`        | string       | Unique process identifier (PID or container ID).                            | `12345`                           |
| `thread_id`         | string       | Thread identifier for multi-threaded apps.                                  | `thread-1`                        |
| `message`           | string       | Human-readable log content.                                                  | `"Failed to validate token."`     |
| `trace_id`          | string       | Correlation ID for distributed tracing.                                     | `abc123-xyz456`                   |
| `stack_trace`       | string       | Error traceback (if applicable).                                             | `Null` (unless ERROR).            |
| `metadata`          | object       | Key-value pairs (e.g., `user_id`, `request_id`).                            | `{"user_id": "user123"}`          |

### **2. Database Log Schema**
| Field               | Type         | Description                                                                 | Example Value                     |
|---------------------|--------------|-----------------------------------------------------------------------------|-----------------------------------|
| `timestamp`         | datetime     | Log generation time.                                                        | `2024-05-15T14:35:20.000Z`       |
| `level`             | string       | Severity (e.g., `ERROR`, `NOTICE`).                                         | `ERROR`                           |
| `source`            | string       | Database system (e.g., `postgresql`, `mongodb`).                            | `postgresql`                      |
| `query`             | string       | SQL/NoSQL query executed.                                                  | `SELECT * FROM users WHERE id = ?`|
| `status`            | string       | Query outcome (e.g., `SUCCESS`, `FAILURE`).                                 | `FAILURE`                         |
| `duration_ms`       | integer      | Query execution time.                                                       | `1200`                            |
| `error`             | string       | Error message (if applicable).                                              | `Timeout during connection.`      |
| `connection_id`     | string       | Unique connection identifier.                                               | `conn-789`                        |

### **3. Server Log Schema (e.g., Nginx)**
| Field               | Type         | Description                                                                 | Example Value                     |
|---------------------|--------------|-----------------------------------------------------------------------------|-----------------------------------|
| `timestamp`         | datetime     | Log entry time.                                                            | `2024-05-15T14:40:10.00Z`         |
| `level`             | string       | Severity (e.g., `info`, `warn`, `error`).                                  | `error`                           |
| `host`              | string       | Server hostname/IP.                                                        | `web-server-01`                   |
| `client_ip`         | string       | Requesting client’s IP.                                                    | `192.168.1.100`                   |
| `method`            | string       | HTTP method (GET, POST, etc.).                                             | `POST`                            |
| `path`              | string       | Requested endpoint.                                                        | `/api/users`                      |
| `status`            | integer      | HTTP status code (e.g., `404`, `500`).                                     | `500`                             |
| `response_time_ms`  | integer      | Time taken to respond.                                                     | `3500`                            |
| `user_agent`        | string       | Client’s browser/device info.                                              | `Mozilla/5.0`                    |

## **Query Examples**
### **1. Filtering Errors by Component**
**Goal**: Find all `ERROR` logs from the `payment-gateway` in the last 24 hours.
**Query (ELK/Kibana)**:
```json
{
  "query": {
    "bool": {
      "must": [
        { "match": { "timestamp": { "range": { "gte": "now-24h" } } } },
        { "term": { "level": "ERROR" } },
        { "term": { "component": "payment-gateway" } }
      ]
    }
  }
}
```
**Query (SQL-like, e.g., Splunk)**:
```sql
index=application_svc
| search level=ERROR component="payment-gateway"
| stats count by message
| sort -count
```

---

### **2. Correlating Failed Requests with Database Errors**
**Goal**: Identify which API requests caused database timeouts (correlation via `trace_id`).
**Steps**:
1. **Filter Database Errors**:
   ```json
   {
     "query": {
       "bool": {
         "must": [
           { "match": { "source": "postgresql" } },
           { "match_phrase": { "error": "Timeout" } }
         ]
       }
     }
   }
   ```
2. **Extract `trace_id`** from error logs (e.g., `trace_id: xyz789`).
3. **Query Application Logs**:
   ```json
   {
     "query": {
       "bool": {
         "must": [
           { "term": { "trace_id": "xyz789" } },
           { "match": { "level": "ERROR" } }
         ]
       }
     }
   }
   ```
4. **Visualize Timeline**: Use tools like Grafana or ELK’s "Timeline" to see request flow.

---

### **3. Performance Bottleneck Analysis**
**Goal**: Detect slow API endpoints (response time > 2s).
**Query (Prometheus + Grafana)**:
```promql
# Latency histogram (e.g., 99th percentile)
histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket[5m])) by (le, endpoint))
```
**Log Query (ELK)**:
```json
{
  "query": {
    "range": {
      "field": "timestamp",
      "from": "now-1h",
      "to": "now"
    }
  },
  "aggs": {
    "slow_requests": {
      "terms": {
        "field": "path",
        "order": { "avg_response_time": "desc" }
      },
      "aggs": {
        "avg_response_time": { "avg": { "field": "response_time_ms" } }
      }
    }
  }
}
```

### **4. User-Specific Issues**
**Goal**: Find all logs for a specific user (`user_id: 456`).
**Query**:
```json
{
  "query": {
    "bool": {
      "must": [
        { "term": { "metadata.user_id": "456" } }
      ]
    }
  }
}
```
**Filter by Combined Fields (e.g., `ERROR` + `user_id`)**:
```sql
index=app_logs
| search level=ERROR metadata.user_id="456"
| table _timestamp, component, message
```

### **5. Alerting on Anomalies**
**Goal**: Set up alerts for repeated `500` errors on `/checkout` endpoint.
**ELK Alert (Lens/Watch)**:
```json
{
  "condition": {
    "script": {
      "source": """
        emit {
          "count": docs.count > 5,
          "message": "High error rate on /checkout"
        }
      """,
      "lang": "painless"
    },
    "filter": {
      "bool": {
        "must": [
          { "term": { "path": "/checkout" } },
          { "term": { "status": "500" } }
        ]
      }
    }
  }
}
```

## **Troubleshooting Workflow**
1. **Reproduce the Issue**:
   - Confirm whether the problem is intermittent or consistent.
   - Collect logs from multiple environments (dev/stage/prod).

2. **Filter Logs**:
   - Use time ranges (e.g., "last hour").
   - Narrow by `component`, `level`, or `trace_id`.

3. **Analyze Patterns**:
   - Check for repeated errors, spikes, or missing logs.
   - Correlate with external events (e.g., database migrations).

4. **Investigate Root Cause**:
   - **Application Errors**: Review stack traces and metadata (e.g., `user_id`).
   - **Database Issues**: Check slow queries or connection pool exhaustion.
   - **Dependency Failures**: Look for external API timeouts.

5. **Resolve and Validate**:
   - Apply fixes (e.g., code updates, scaling).
   - Verify resolution by monitoring logs post-change.

6. **Prevent Recurrence**:
   - Add logging for edge cases.
   - Set up proactive alerts (e.g., `ERROR` rate > 10%/min).

## **Common Pitfalls & Solutions**
| Pitfall                          | Solution                                                                 |
|-----------------------------------|--------------------------------------------------------------------------|
| **Log Volume Overwhelm**          | Implement log retention policies (e.g., 30 days) or use sampling.          |
| **Unstructured Logs**             | Enforce structured logging (JSON) for easier querying.                   |
| **Missing Correlation IDs**       | Add `trace_id` or `request_id` to all relevant logs.                      |
| **False Positives in Alerts**     | Tune alert thresholds (e.g., only alert on `CRITICAL` + repeated errors).|
| **Delayed Log Shipping**          | Ensure logs are buffered and flushed periodically (e.g., every 10s).     |
| **Over-Logging**                  | Use `DEBUG` sparingly; default to `INFO` in production.                   |

## **Related Patterns**
1. **[Distributed Tracing]**
   - Use `trace_id` in logs to correlate requests across services (e.g., OpenTelemetry).
   - *Tools*: Jaeger, Zipkin.

2. **[Metrics-Driven Observability]**
   - Combine logs with metrics (e.g., latency, error rate) for deeper analysis.
   - *Tools*: Prometheus + Grafana, Datadog.

3. **[Structured Event Sourcing]**
   - Store logs as immutable event streams for auditability.
   - *Use Case*: Financial transactions, compliance logging.

4. **[Chaos Engineering for Logging]**
   - Test log reliability by simulating failures (e.g., disk full errors).
   - *Tools*: Gremlin, Chaos Monkey.

5. **[Log Aggregation & Indexing]**
   - Centralize logs for scalable querying (e.g., ELK, Loki).
   - *Optimization*: Use log sharding or time-based indexing.

6. **[Security Logging]**
   - Focus on `AUTHENTICATION_FAILED`, `UNAUTHORIZED_ACCESS`, or `SQL_INJECTION` patterns.
   - *Tools*: SIEM (Splunk, Wazuh).

7. **[Performance Profiling]**
   - Use logs to identify slow methods or I/O bottlenecks.
   - *Query*: Filter by `duration_ms > threshold`.

8. **[Canary Deployments with Log Analysis]**
   - Monitor new feature rollouts via log correlation IDs.
   - *Example*: Compare `trace_id` distributions pre/post-deploy.
