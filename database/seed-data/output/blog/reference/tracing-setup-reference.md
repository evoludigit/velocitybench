# **[Pattern] Tracing Setup – Reference Guide**

---

## **Overview**
The **Tracing Setup** pattern ensures observability by capturing and correlating distributed requests across microservices. It enables tracing workflows, debugging performance bottlenecks, and analyzing user journeys. This guide covers implementation requirements, schema structures, query examples, and best practices for setting up distributed tracing efficiently.

---

## **Implementation Details**

### **Key Concepts**
1. **Tracer Provider**: Generates tracing data (spans, events) and sends them to a backend (e.g., Jaeger, OpenTelemetry Collector).
2. **Span**: Represents a single operation (e.g., API call, database query) with metadata (start/end time, attributes).
3. **Trace**: A collection of correlated spans forming a distributed request flow.
4. **Sampler**: Decides whether to trace a request (e.g., `AlwaysOn`, `Probabilistic`).
5. **Export Processors**: Formats and sends data to backends (e.g., Jaeger, Zipkin).

---

## **Schema Reference**

### **Tracing Setup Schema**
| **Component**       | **Description**                                                                 | **Example**                                                                 |
|----------------------|---------------------------------------------------------------------------------|-----------------------------------------------------------------------------|
| **Tracer Provider**  | Initializes tracing with SDK (e.g., OpenTelemetry).                            | `TraceProvider provider = Tracing.getTracerProvider();`                     |
| **Sampler**          | Configures sampling logic (`AlwaysOn`, `ParentBased` with `TraceIdRatio`).     | `Sampling.DynamicSamplingStrategy strategy = DynamicSamplingStrategy.fromConfig();` |
| **Span**             | Represents an operation with metadata.                                         | `Span span = tracer.spanBuilder("database.query").startSpan();`            |
| **Trace**            | Aggregates correlated spans into a single workflow.                             | `TraceId traceId = "12345-67890"` (auto-generated by the tracer).          |
| **Export Processor** | Sends spans to a backend (e.g., Jaeger exporter, OpenTelemetry Collector).     | `BatchingSpanProcessor processor = BatchingSpanProcessor.builder(...)      |
| **Backend**          | Stores and visualizes traces (e.g., Jaeger UI, Grafana).                     | `JaegerExporter.builder().setEndpoint(...)                                 |
| **Context Propagation** | Attaches trace context (e.g., `TraceId`) to HTTP headers for correlation.  | `W3CFormat.getGlobalTextMapPropagator().inject(textMap, spanContext);`     |

## **Implementation Steps**
1. **Initialize the Tracer Provider**
   ```java
   // Spring Boot (OpenTelemetry)
   @Bean
   public TracerProvider tracerProvider() {
       SdkTracerProvider provider = SdkTracerProvider.builder()
           .addSpanProcessor(BatchingSpanProcessor.builder(JaegerExporter.create()).build())
           .setSampler(Sampler.traceIdRatioBased(0.1))
           .build();
       OTTelemetrySdk.init(TelemetrySdk.builder().setTracerProvider(provider).build());
       return provider;
   }
   ```

2. **Start a Span and Record Attributes**
   ```java
   try (Span span = tracer.spanBuilder("user.login").startSpan()) {
       span.setAttribute("user.id", "12345");
       span.addEvent("auth.check", new Attributes().put("status", "success"));
       // Business logic executes here...
   }
   ```

3. **Correlate Spans via Context Propagation**
   ```java
   // Inject trace context into HTTP headers
   W3CFormat.getGlobalTextMapPropagator().inject(
       new HttpServerRequestHeadersAdapter(request),
       span.getSpanContext()
   );
   ```

4. **Export Data to Backend**
   - Configure a **batching processor** (reduces overhead):
     ```java
     BatchingSpanProcessor processor = BatchingSpanProcessor.builder(JaegerExporter.create())
         .setScheduleDelay(Duration.ofSeconds(5))
         .build();
     provider.addSpanProcessor(processor);
     ```

## **Query Examples**
### **1. Querying Jaeger UI**
   - **Filter by Trace ID**:
     ```
     TraceID: 12345-67890
     ```
   - **Find slow spans**:
     ```
     Duration: > 500ms
     Service: "payment-service"
     ```

### **2. OpenTelemetry Query (Prometheus)**
   ```promql
   # Latency percentiles
   histogram_quantile(0.95, rate(service_latency_bucket[5m]))

   # Error-rate per service
   sum(rate(service_span_errors_total[5m]))
     by (service_name) / sum(rate(service_span_total[5m]))
     by (service_name)
   ```

### **3. Log Correlation with Traces**
   ```bash
   # Link logs to a trace (ELK Stack)
   curl -XPOST http://logstash:9600/_bulk \
     -H 'Content-Type: application/x-ndjson' \
     -d '[{"trace_id":"12345"}, {"message":"Login failed"}]'
   ```

## **Best Practices**

### **Performance**
- **Sampling**: Use `TraceIdRatioBased` (e.g., 10%) to reduce overhead.
- **Batch Processing**: Enable buffering (e.g., `BatchingSpanProcessor`) to minimize I/O calls.
- **Context Propagation**: Ensure all outbound calls (HTTP, gRPC) carry the trace context.

### **Data Management**
- **Retention**: Configure backend (e.g., Jaeger) to retain traces for 7–30 days.
- **Anonymization**: Mask PII (e.g., `user.email`) before exporting.

### **Error Handling**
- **Deadlines**: Set explicit span deadlines (e.g., `Context.current().withDeadline(Duration.ofSeconds(30))`).
- **Circuit Breaking**: Combine with resilience patterns (e.g., Retry/Timeout).

## **Common Pitfalls**
| **Issue**               | **Solution**                                                                 |
|--------------------------|-----------------------------------------------------------------------------|
| **No traces in backend** | Verify sampler is enabled and exporter is active.                           |
| **High latency overhead**| Reduce sampling rate or use `AlwaysOff` for dev environments.               |
| **Missing context**      | Ensure `Context.current()` is propagated across service boundaries.         |
| **Duplicate traces**     | Use unique `TraceID` generation (e.g., UUID).                              |

---

## **Related Patterns**
1. **[Resilience Pattern]** – Combine with retry/circuit breaker to handle failures with context.
2. **[Observability Pattern]** – Augment with metrics (Prometheus) and logs (ELK) for holistic monitoring.
3. **[Context Propagation]** – Standardize headers (e.g., `X-B3-TraceId`) for traceability.
4. **[Distributed Locks]** – Use trace IDs to correlate lock acquisition/release events.
5. **[Rate Limiting]** – Apply limits per trace to prevent abuse in microservices.
