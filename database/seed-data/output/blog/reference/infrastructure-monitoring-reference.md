# **[Pattern] Infrastructure Monitoring Reference Guide**

---
## **1. Overview**
The **Infrastructure Monitoring** pattern is a structured approach to observing, collecting, and analyzing telemetry data from servers, networks, containers, cloud resources, and on-premises hardware. This pattern ensures operational visibility, allowing teams to detect performance anomalies, failures, and security threats proactively. It integrates monitoring tools (e.g., Prometheus, Grafana, Zabbix), logging systems (e.g., ELK, Splunk), and observability platforms (e.g., Datadog, New Relic) to create a unified, scalable monitoring solution.

Key benefits include:
- **Proactive Issue Detection** – Alerts on degradation before outages impact users.
- **Performance Optimization** – Identifies bottlenecks and resource inefficiencies.
- **Compliance & Auditing** – Tracks infrastructure health for regulatory reporting.
- **Cost Efficiency** – Prevents over-provisioning by right-sizing resources.

This guide covers **core components**, **data collection**, **alerting**, and **best practices** for implementing a robust monitoring strategy.

## **2. Schema Reference**

Below is a reference schema for Infrastructure Monitoring, detailing key entities, their attributes, and relationships.

| **Entity**            | **Description**                                                                 | **Attributes**                                                                 | **Relationships**                          |
|-----------------------|-------------------------------------------------------------------------------|-------------------------------------------------------------------------------|--------------------------------------------|
| **Monitoring Agent**  | Software process collecting metrics, logs, and traces from a host.            | Agent ID, Hostname, OS, Version, Health Status, Configuration File Path       | Deployed on (InfrastructureResource)       |
| **InfrastructureResource** | A physical/virtual component under monitoring (e.g., server, VM, container).   | Resource ID, Type (e.g., `Server`, `Container`, `CloudVM`), IP Address, CPU, RAM, Disk Usage, Uptime, Status | Hosts (MonitoringAgent), Monitors (Metric) |
| **Metric**            | Quantitative data points (e.g., CPU usage, latency).                          | Metric Name (e.g., `cpu_usage`), Value, Timestamp, Unit (e.g., `%`, `ms`), Thresholds (e.g., Warn/Alert) | Belongs to (InfrastructureResource), Collected by (MonitoringAgent) |
| **Log**               | Structured/unstructured event data from applications, services, or infrastructure. | Log ID, Timestamp, Level (e.g., `INFO`, `ERROR`), Message, Severity, Source | Generated by (InfrastructureResource)      |
| **Trace**             | Distributed tracing data for performance bottlenecks (e.g., microservices).   | Trace ID, Span ID, Operation Name, Timestamps (start/end), Duration, Tags     | Related to (InfrastructureResource)        |
| **Alert Rule**        | Conditions triggering notifications (e.g., CPU > 90% for 5 minutes).          | Rule ID, Name, Condition (e.g., `metric > threshold`), Severity (e.g., `Critical`), Notification Channels (e.g., Slack, PagerDuty) | Triggered by (Metric/Log/Trace)           |
| **Notification**      | User alert (e.g., email, SMS, chatbot message).                              | Notification ID, Status (e.g., `Acknowledged`, `Resolved`), Timestamp, Recipient, Summary, Details | Sent via (Alert Rule)                     |
| **Dashboard**         | Visual representation of monitored metrics/logs (e.g., Prometheus + Grafana).  | Dashboard ID, Title, Widgets (e.g., CPU Graph, Log Table), Time Range          | Displays (Metric/Log/Trace)               |
| **Configuration**     | Centralized settings for monitoring tools (e.g., Prometheus scrape intervals).  | Config ID, Tool (e.g., `Prometheus`, `Grafana`), Parameters (e.g., `scrape_interval: 15s`), Version | Applied to (MonitoringAgent)              |

## **3. Query Examples**

### **3.1 Monitor Resource Health (SQL-like Pseudocode)**
```sql
-- Check servers with high CPU load (>80%) in the last hour
SELECT
    resource_id,
    hostname,
    metric_name,
    value,
    timestamp
FROM metrics
WHERE
    metric_name = 'cpu_usage'
    AND value > 80
    AND timestamp > NOW() - INTERVAL '1 hour'
ORDER BY value DESC;
```

### **3.2 Alert on Failed Containers (PromQL)**
```promql
# Alert if container restart count exceeds 3 in 5 minutes
up{job="kubernetes-pods"} == 0
or
kube_pod_container_status_restarts_total{job="kubernetes-pods"} > 3
[5m]
```

### **3.3 Aggregate Log Errors (ELK Query DSL)**
```json
{
  "query": {
    "bool": {
      "must": [
        { "match": { "level": "ERROR" } },
        { "range": { "@timestamp": { "gte": "now-1h" } } }
      ]
    }
  }
}
```

### **3.4 Trace Latency Analysis (OpenTelemetry Query)**
```sql
-- Find traces where HTTP response time > 200ms
SELECT
    trace_id,
    operation_name,
    avg(duration)
FROM traces
WHERE
    operation_name LIKE '%HTTP%'
    AND duration > 200
GROUP BY trace_id, operation_name;
```

## **4. Implementation Best Practices**

### **4.1 Data Collection**
- **Agent vs. Agentless**:
  - Use **agents** (e.g., Prometheus Node Exporter, Datadog Agent) for granular host-level metrics.
  - Use **agentless** (e.g., cloud provider APIs, SNMP) for cloud resources or legacy systems.
- **Sampling**: Reduce volume with statistical sampling (e.g., collect metrics every 10s instead of 1s).
- **Data Retention**: Retain critical metrics long-term (e.g., 30 days) but purge high-volume logs after 7 days.

### **4.2 Alerting Strategies**
- **Severity Levels**:
  - `Critical` (e.g., disk full, database crash) → Immediate action.
  - `Warning` (e.g., high latency) → Monitor.
  - `Informational` (e.g., service restarted) → Review later.
- **Alert Fatigue**: Set reasonable thresholds and use **noise reduction** (e.g., ignore spurious alerts).
- **Notification Channels**: Combine email (for summaries), Slack (for real-time), and PagerDuty (for critical issues).

### **4.3 Dashboard Design**
- **Focus on SLOs**: Design dashboards around **Service-Level Objectives** (e.g., "99.9% uptime").
- **Visual Hierarchy**: Prioritize critical metrics (e.g., uptime, error rates) over secondary data.
- **Dynamic Time Ranges**: Allow users to switch between short-term (5m) and long-term (30d) views.

### **4.4 Security**
- **Data Encryption**: Encrypt metrics in transit (TLS) and at rest.
- **Access Control**: Restrict dashboard access by role (e.g., DevOps vs. Dev).
- **Audit Logs**: Log access to monitoring tools for compliance.

### **4.5 Scalability**
- **Distributed Agents**: Use tools like **Prometheus Relay** or **VictoriaMetrics** for large-scale deployments.
- **Time-Series Database**: Choose databases optimized for metrics (e.g., InfluxDB, TimescaleDB).
- **Multi-Cloud**: Deploy agents in each cloud provider’s region to avoid latency.

## **5. Related Patterns**

| **Pattern**               | **Description**                                                                 | **Use When**                                  |
|---------------------------|-------------------------------------------------------------------------------|-----------------------------------------------|
| **Logging**                | Centralized collection, storage, and analysis of logs.                         | Need structured event data for debugging.    |
| **Tracing**                | Tracking requests across distributed services to identify bottlenecks.        | Microservices architecture.                   |
| **Auto-Remediation**       | Automatically fixing issues detected by monitoring (e.g., restarting pods).   | Critical systems requiring zero downtime.    |
| **Chaos Engineering**      | Testing system resilience by simulating failures.                             | Proactively validate monitoring and recovery. |
| **Configuration Management** | Managing monitoring tool configurations (e.g., Prometheus targets).      | Large-scale, multi-environment deployments.  |
| **Observability as Code**  | Defining monitoring dashboards, alerts, and rules in code (e.g., Terraform).  | Infrastructure-as-Code (IaC) workflows.       |

## **6. Glossary**
| **Term**               | **Definition**                                                                 |
|------------------------|---------------------------------------------------------------------------------|
| **SLO**                | Service Level Objective: Quantitative goal (e.g., "99.9% availability").         |
| **SLI**                | Service Level Indicator: Metric used to measure SLO (e.g., error rate).          |
| **APM**                | Application Performance Monitoring: Focuses on application-level metrics.       |
| **ITSM**               | IT Service Management: Links monitoring to ITIL processes (e.g., incident mgmt).|
| **Agentless Monitoring**| Monitoring without installing software on targets (e.g., cloud APIs).         |
