# **[Pattern] Signing Verification – Reference Guide**

---

## **1. Overview**
The **Signing Verification** pattern ensures data integrity and authenticity by validating cryptographic signatures attached to a payload. It is critical in scenarios where unverified data could lead to security breaches, fraud, or unauthorized access. This pattern applies to REST APIs, event-driven systems, IoT devices, and blockchain interactions.

Signing verification uses **asymmetric cryptography** (e.g., RSA, ECDSA) or **symmetric hashing** (e.g., HMAC) to confirm:
- **Authenticity** – The data was sent by a trusted source.
- **Integrity** – The data has not been tampered with.

This guide covers key concepts, schema requirements, implementation checks, and integration with related patterns.

---

## **2. Key Concepts**
| **Term**               | **Definition**                                                                                     | **Example**                          |
|-------------------------|---------------------------------------------------------------------------------------------------|--------------------------------------|
| **Payload**             | The data being signed and verified. Can include headers, body, or a full request.               | JSON API response or IoT sensor data |
| **Signature**           | A cryptographic token (base64-encoded) generated by signing the payload with a private key.      | `"d5e7a4b3..."`                      |
| **Signature Algorithm** | The cryptographic method used (e.g., `HS256`, `RS256`).                                           | `ES256` for elliptic curve signatures |
| **Public Key**          | Used by the verifier to check the signature’s validity. Cannot derive the private key.           | PEM or JWK format                  |
| **Timestamp**           | Prevents replay attacks by indicating when the payload was signed.                                | `2024-03-20T14:30:00Z`              |
| **Nonce**               | A unique value to ensure freshness (if replay protection is required).                           | `"a1b2c3d4-..."`                    |

## **3. Schema Reference**
### **Request/Response Structure**
```json
{
  "payload": {
    "@type": "string", // MIME type of the payload (e.g., "application/json")
    "data": "..."      // The actual payload (can be any format)
  },
  "signature": {
    "algorithm": "ES256", // Signature algorithm
    "value": "d5e7a4b3...", // Base64-encoded signature
    "keyId": "abc123",       // Identifier for the public key
    "timestamp": "2024-03-20T14:30:00Z", // Signing timestamp
    "nonce": "a1b2c3d4-...", // Optional replay protection
    "expiresAt": "2024-03-21T14:30:00Z" // Optional expiry
  }
}
```

### **Public Key Schema**
```json
{
  "keys": [
    {
      "kid": "abc123", // Key identifier (must match `signature.keyId`)
      "key": {
        "kty": "EC",         // Key type (e.g., "RSA", "EC")
        "crv": "P-256",      // Curve (for EC)
        "x": "AQ...",        // Public key components
        "y": "BC...",
        "alg": "ES256"       // Algorithm
      },
      "use": "sig",         // Key usage (signature)
      "exp": "2024-04-20T00:00:00Z" // Expiry
    }
  ]
}
```

### **Validation Rules**
| **Check**                  | **Purpose**                                                                                     | **Validation**                                                                 |
|----------------------------|------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------|
| **Signature Presence**     | Ensure the signature exists.                                                                 | `req.headers["Signature"]` must exist.                                         |
| **Algorithm Support**      | Verify the algorithm is supported by the verifier.                                              | Check against a whitelist (e.g., `HS256`, `RS256`).                            |
| **Public Key Existence**   | Confirm the public key exists and is active.                                                   | Fetch from JWKS endpoint or local cache.                                       |
| **Payload Integrity**      | Reconstruct the payload to verify the signature.                                               | Use `HMAC-SHA256` or `ECDSA` with payload header.                              |
| **Timestamp Validity**     | Prevent replay attacks by ensuring the signature is recent.                                    | `timestamp > now - 5 minutes`.                                                 |
| **Nonce Uniqueness**       | Ensure the nonce hasn’t been used before (if required).                                         | Check against a local `nonce` store.                                           |
| **Key Expiry**             | Reject expired keys to avoid vulnerabilities.                                                   | `now < key.exp`.                                                              |
| **Key Rotation**           | Handle key rollovers gracefully.                                                               | Maintain multiple active keys in JWKS.                                        |

## **4. Implementation Details**
### **Step 1: Configure Supported Algorithms**
```javascript
const SUPPORTED_ALGORITHMS = {
  "HS256": require("crypto").createHmac("sha256"),
  "RS256": require("crypto").createVerify("RSA-SHA256"),
  "ES256": require("crypto").createVerify("ECDSA-SHA256")
};
```

### **Step 2: Fetch or Cache Public Keys**
```javascript
async function getPublicKey(keyId) {
  const jwksResponse = await fetch("https://example.com/.well-known/jwks.json");
  const jwks = await jwksResponse.json();
  return jwks.keys.find(k => k.kid === keyId);
}
```

### **Step 3: Verify the Signature**
```javascript
async function verifySignature(payload, signatureHeader) {
  const { algorithm, value, keyId } = signatureHeader;
  const publicKey = await getPublicKey(keyId);

  // Sign the payload (e.g., hex-encoded header + payload)
  const signingInput = Buffer.from(
    `(${algorithm})=${signingHeader}`
  );

  // Verify
  const verifier = SUPPORTED_ALGORITHMS[algorithm];
  verifier.update(signingInput).end(Buffer.from(value, "base64"));
  return verifier.verify(publicKey, payload);
}
```

### **Step 4: Handle Replay Attacks (Optional)**
```javascript
const nonceStore = new Set();

function checkNonce(nonce) {
  if (nonceStore.has(nonce)) {
    throw new Error("Replay detected: nonce already used");
  }
  nonceStore.add(nonce);
}
```

## **5. Query Examples**
### **Example 1: Validating an API Response**
**Request Headers:**
```
Content-Type: application/json
Authorization: Bearer <JWT>
Signature: algorithm="ES256", keyId="abc123", timestamp="2024-03-20T14:30:00Z", nonce="a1b2c3d4", value="d5e7a4b3..."
```

**Server Logic:**
```javascript
const response = await fetch("https://api.example.com/data");
const signature = response.headers.get("Signature");
const payload = await response.text();

if (!verifySignature(payload, signature)) {
  throw new Error("Invalid signature");
}
```

### **Example 2: Verifying IoT Device Data**
**Payload:**
```json
{
  "deviceId": "sensor-001",
  "timestamp": "2024-03-20T14:30:00Z",
  "temperature": 22.5,
  "signature": {
    "algorithm": "HS256",
    "value": "xyz123..."
  }
}
```

**Server Logic:**
```javascript
const secretKey = await getIoTSigningKey("sensor-001");
const hmac = crypto.createHmac("sha256", secretKey);
const signature = hmac.update(JSON.stringify(payload)).digest("base64");

if (!compareSignatures(signature, payload.signature.value)) {
  throw new Error("Tampered data");
}
```

## **6. Error Handling**
| **Error Type**            | **Code** | **Response**                                                                 |
|---------------------------|----------|-----------------------------------------------------------------------------|
| **Missing Signature**     | 401      | `{"error": "Signature header is required"}`                               |
| **Unsupported Algorithm** | 400      | `{"error": "Algorithm not supported"}`                                     |
| **Invalid Signature**     | 403      | `{"error": "Invalid signature"}`                                           |
| **Expired Key**           | 401      | `{"error": "Public key expired"}`                                           |
| **Replay Attack**         | 409      | `{"error": "Nonce already used"}`                                           |
| **Key Not Found**         | 404      | `{"error": "Public key not found"}`                                         |

## **7. Performance Considerations**
- **Cache Public Keys**: Avoid fetching keys on every request (TTL: 1 hour).
- **Parallel Verification**: Use worker threads for CPU-heavy operations (e.g., RSA).
- **Precompute Hashes**: For HMAC, precompute hashes when possible.
- **Algorithm Selection**: Prefer **ECDSA** over RSA for lower latency.

## **8. Related Patterns**
| **Pattern**               | **Description**                                                                                     | **Integration Point**                                                                 |
|---------------------------|---------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------|
| **[JWT (JSON Web Tokens)](https://tools.ietf.org/html/rfc7519)** | Uses signatures for authentication/authorization.                                               | Replace or complement `Signature` header with JWT.                                    |
| **[HMAC-Based Authentication](https://datatracker.ietf.org/doc/html/rfc7693)** | Symmetric signing for APIs with shared secrets.                                                  | Use `HS256`/`HS512` if both parties share a key.                                       |
| **[Digital Certificates](https://www.rfc-editor.org/rfc/rfc5280)** | Public-key infrastructure (PKI) for key distribution.                                             | Use certificates to validate public keys instead of JWKS.                                |
| **[Key Rotation](https://aws.amazon.com/blogs/security/key-rotation-best-practices/)**          | Automatically rotate keys to reduce exposure.                                                    | Implement JWKS endpoints with `exp` fields and auto-rotation scripts.                    |
| **[Event Sourcing](https://martinfowler.com/eaaCatalog/eventSourcing.html)**                  | Audit signed events for auditability.                                                           | Append signatures to event payloads for replay protection and consistency checks.       |
| **[Zero Trust](https://www.zero-trust-networking.com/)**                                     | Verify every request, even from internal systems.                                               | Combine signing verification with mutual TLS (mTLS) for additional security.           |

## **9. Best Practices**
1. **Use Short-Lived Keys**: Rotate keys every 24–72 hours.
2. **Validate Headers First**: Check `Signature` before processing payloads.
3. **Log Signature Failures**: Monitor failed verifications for anomalies.
4. **Rate-Limit Verification**: Prevent brute-force attacks on signature checks.
5. **Document Supported Algorithms**: Transparently list accepted algorithms.
6. **Test Offline Signing**: Simulate key compromise scenarios.
