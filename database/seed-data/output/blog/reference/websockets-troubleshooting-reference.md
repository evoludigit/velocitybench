# **[Pattern] Websockets Troubleshooting Reference Guide**

---

## **Overview**
WebSockets provide full-duplex, real-time communication between a client and server over a single TCP connection, eliminating the latency and overhead of traditional HTTP polling mechanisms. When issues arise—such as connection drops, error messages, or latency—they can disrupt critical applications like chat platforms, live updates, or collaborative tools. This guide provides a structured approach to diagnosing and resolving WebSocket-related problems, covering common failure points, debugging techniques, and best practices for maintaining reliable connections.

---

## **Key Concepts & Implementation Details**

### **1. WebSocket Lifecycle Phases**
A WebSocket connection follows these phases:
1. **Handshake** – Client initiates with an HTTP upgrade request; server responds with `101 Switching Protocols`.
2. **Open** – Active bidirectional communication.
3. **Frame Exchange** – Data is sent/received via WebSocket frames.
4. **Close** – Clean or abrupt termination via `Connection Close` frame.

**Common Failure Points:**
| Phase          | Issue                          | Root Cause                                                                 |
|----------------|--------------------------------|----------------------------------------------------------------------------|
| **Handshake**  | `400 Bad Request`               | Incorrect `Sec-WebSocket-Key` or `Upgrade` header.                       |
| **Open**       | Connection drops (`1000` code) | Network instability, server limits, or client-side timeouts.               |
| **Frame**      | Corrupted data                 | Fragmentation errors, unsupported masks, or malformed messages.           |
| **Close**      | Hanging connection             | Server not processing `Close` frames; client not releasing resources.    |

---

### **2. Error Codes & Messages**
WebSocket errors are defined by [RFC 6455](https://datatracker.ietf.org/doc/html/rfc6455#section-7.4.1). Key codes:
- **1000** – Normal closure (successful).
- **1001** – Endpoint going away (graceful shutdown).
- **1002** – Protocol error (e.g., malformed frames).
- **1003** – Unsupported data (e.g., binary vs. text mismatch).
- **1006** – Abnormal closure (no status code).

**Debugging Tip:**
- Log **raw frames** (e.g., using Wireshark or browser DevTools).
- Check server logs for `OnClose` events with custom status codes.

### **3. Common Troubleshooting Scenarios**
#### **Scenario 1: Connection Refuses to Establish**
- **Symptoms:** Client stuck on `CONNECTING`, server logs show no handshake.
- **Diagnosis:**
  - Verify `Sec-WebSocket-Key` is base64-encoded and correctly hashed.
  - Check CORS policies if client and server are cross-origin.
  - Ensure no firewall/NAT blocks port `80`/`443` (or custom WebSocket port).
- **Fix:**
  ```bash
  # Test handshake manually (cURL)
  curl -v -H "Upgrade: websocket" -H "Connection: Upgrade" -H "Sec-WebSocket-Key: [key]" http://example.com/endpoint
  ```

#### **Scenario 2: Frequent Drops (1006)**
- **Symptoms:** Messages arrive out of order or incomplete.
- **Diagnosis:**
  - **Latency:** Check RTT (Round-Trip Time) via `performance.now()` timestamps.
  - **Network:** Use `mtr` or `traceroute` to identify packet loss.
  - **Server:** Monitor CPU/memory usage for resource exhaustion.
- **Fix:**
  - Implement **reconnection logic** (exponential backoff).
  - Use **ping/pong frames** (`Opcode 9/10`) to detect dead connections.

#### **Scenario 3: Binary vs. Text Mismatch (1003)**
- **Symptoms:** Server rejects messages with `1003`.
- **Diagnosis:**
  - Verify `Accept` header in handshake matches key.
  - Ensure frames are sent as **binary** or **text** consistently.
- **Fix:**
  ```javascript
  // Client: Explicitly set text mode
  const socket = new WebSocket('ws://example.com', {
    protocols: ['text'],
  });

  // Server: Validate frame type (Node.js example)
  socket.on('message', (data) => {
    if (!Buffer.isBuffer(data)) { // Force binary if needed
      data = Buffer.from(data);
    }
  });
  ```

## **Schema Reference**
| Component          | Field/Parameter               | Type               | Description                                                                 |
|--------------------|-------------------------------|--------------------|-----------------------------------------------------------------------------|
| **Handshake Headers** | `Sec-WebSocket-Key`          | Base64-encoded     | Generated by client; server must hash with `SHA-1` and `GUID` (`258EAFA5`). |
|                     | `Sec-WebSocket-Extensions`    | String             | Optional (e.g., `permessage-deflate`).                                       |
|                     | `Upgrade`                     | String             | Must be `websocket`.                                                        |
| **Frame Structure** | `FIN` (Bit 0)                 | Boolean           | Indicates last frame in sequence.                                           |
|                     | `RSV1/RSV2/RSV3` (Bits 4-6)   | Boolean           | Reserved; must be `0` (unmasked server frames).                            |
|                     | `Opcode`                      | 4-bit             | `0x1` (text), `0x2` (binary), `0x8` (close), `0x9` (ping).                 |
| **Error Handling**  | `Close Code`                  | 16-bit integer    | Standard or custom (e.g., `4001` for "Retry").                               |
| **Reconnection**    | `reconnectInterval`           | Number (ms)        | Default: `1000`; exponential backoff: `1000 * 2^n`.                         |

## **Query Examples**
### **1. Verify WebSocket Connection (cURL)**
Test handshake with full headers:
```bash
curl -v -X GET "HTTP://example.com/ws" \
  -H "Host: example.com" \
  -H "Upgrade: websocket" \
  -H "Connection: Upgrade" \
  -H "Sec-WebSocket-Key: "dGhlIHNhbXBsZSBub25jZQ==" \
  -H "Sec-WebSocket-Version: 13"
```

### **2. Debug Frame Level (Wireshark Filter)**
Capture WebSocket traffic:
```
tcp.port == 80 && http.request.method == "GET" && http.host contains "example.com" && http.header contains "Upgrade: websocket"
```

### **3. Client-Side Reconnection Logic (JavaScript)**
```javascript
let retryCount = 0;
const maxRetries = 5;
const ws = new WebSocket('wss://example.com');

ws.onopen = () => console.log('Connected');
ws.onclose = () => {
  if (retryCount < maxRetries) {
    const delay = 1000 * Math.pow(2, retryCount);
    setTimeout(() => ws.connect(), delay);
    retryCount++;
  }
};
```

### **4. Server-Side Frame Validation (Node.js)**
```javascript
const WebSocket = require('ws');
const wss = new WebSocket.Server({ port: 8080 });

wss.on('connection', (ws) => {
  ws.on('message', (data) => {
    if (typeof data === 'string') {
      // Process text
    } else if (Buffer.isBuffer(data)) {
      // Process binary
    } else {
      ws.close(1003, 'Unsupported data type');
    }
  });
});
```

## **Best Practices**
1. **Logging:**
   - Log **timestamps** for handshake/close events.
   - Include **client IP** and **user agent** for debugging.
2. **Performance:**
   - **Compress frames** (e.g., `permessage-deflate`) for large payloads.
   - **Batch messages** to reduce overhead.
3. **Security:**
   - Use **WSS (wss://)** with TLS certificates.
   - Validate **origin** headers to prevent CSRF.
4. **Testing:**
   - Simulate **network degradation** (e.g., `tc qdisc` for packet loss).
   - Test **reconnection** under load (e.g., with Locust).
