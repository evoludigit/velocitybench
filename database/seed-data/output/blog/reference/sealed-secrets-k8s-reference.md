# **[Pattern] Sealed Secrets Kubernetes Integration Reference Guide**

---

## **Overview**
This reference guide details **Sealed Secrets integration patterns** for Kubernetes, a tool that encrypts Kubernetes Secrets and allows them to be version-controlled and securely deployed across clusters. Sealed Secrets ensures secrets are encrypted at rest using a **private/encrypted key** (stored in a **SealedSecret resource**) instead of plaintext values, mitigating risks from accidental leaks in configuration repositories.

Key use cases include:
- Secure deployment of secrets across multiple clusters (dev, staging, prod).
- Compliance with **zero-trust principles** by encrypting secrets before they enter Git.
- Integration with **CI/CD pipelines** for immutable deployment workflows.
- Rotation of secrets without redeploying applications.

This guide covers **core concepts, implementation scenarios, schema references, and anti-patterns**.

---

## **Key Concepts**

| Concept | Description |
|---------|------------|
| **Kubernetes `Secret`** | Handles storage of sensitive data (e.g., passwords, tokens) in encrypted form in-cluster. |
| **SealedSecret** | A custom resource that holds the **encrypted version of a `Secret`** (stored in Git). Decrypted in-cluster using a trusted key. |
| **Secrets Encryption Key** | A key generated by the **Kubernetes API server** (used for in-cluster decryption). Must be stored securely. |
| **Sealed Secrets Controller** | A Kubernetes operator that decrypts `SealedSecret` and generates a `Secret`. Handles reconciliation. |
| **External Key Management** | Encryption keys can be managed via cloud providers (AWS KMS, GCP KMS) or HashiCorp Vault for added security. |

## **Implementation Scenarios**

### **1. Basic Integration (Self-Managed Encryption)**
Deploy Sealed Secrets to decrypt secrets using the **native Kubernetes encryption key**.

#### **Steps:**
1. **Install the Sealed Secrets Operator**
   ```sh
   kubectl apply -f https://github.com/bitnami-labs/sealed-secrets/releases/download/v0.26.0/controller.yaml
   ```
   *(Choose the latest version from [releases](https://github.com/bitnami-labs/sealed-secrets/releases).)*

2. **Create a `SealedSecret`**
   Encrypt a `Secret` using:
   ```sh
   kubectl create secret generic my-secret --from-literal=password=MySuperSecret --dry-run=client -o yaml | kubeseal -o yaml > sealed-secret.yaml
   ```
   (Replace `kubeseal` with `sealsecret` if using the newer CLI.)

3. **Deploy the `SealedSecret`**
   ```sh
   kubectl apply -f sealed-secret.yaml
   ```

4. **Verify Decryption**
   The controller will create a `Secret` from the `SealedSecret`:
   ```sh
   kubectl get secret my-secret -o yaml
   ```

#### **Schema Reference**
| Field | Type | Description |
|-------|------|-------------|
| `metadata.name` | string | Name of the `Secret` (e.g., `my-secret`). |
| `spec.template.spec.data` | object | Encrypted key-value pairs (e.g., `password: <encrypted-value>`). |
| `sealedSecretsController.kubernetes.io/autogenerated` | string | Metadata to distinguish auto-generated secrets. |

### **2. External Encryption (Cloud/HashiCorp Vault)**
Use **external key providers** (e.g., AWS KMS) for stronger security.

#### **Steps:**
1. **Configure External Encryption Key**
   Modify the `controller.yaml` to include external provider settings:
   ```yaml
   apiVersion: v1
   kind: ServiceAccount
   metadata:
     name: sealed-secrets-controller
   ---
   apiVersion: rbac.authorization.k8s.io/v1
   kind: ClusterRoleBinding
   metadata:
     name: sealed-secrets-controller-binding
   roleRef:
     apiGroup: rbac.authorization.k8s.io
     kind: ClusterRole
     name: sealed-secrets-controller-role
   subjects:
   - kind: ServiceAccount
     name: sealed-secrets-controller
     namespace: sealed-secrets
   ---
   apiVersion: apps/v1
   kind: Deployment
   metadata:
     name: sealed-secrets-controller
     labels:
       app: sealed-secrets-controller
   spec:
     replicas: 1
     selector:
       matchLabels:
         app: sealed-secrets-controller
     template:
       metadata:
         labels:
           app: sealed-secrets-controller
       spec:
         serviceAccountName: sealed-secrets-controller
         containers:
         - name: controller
           image: bitnami/sealed-secrets-controller:0.26.0
           env:
           - name: STRICT_MODE
             value: "true"
           - name: ENCRYPTION_KEY
             valueFrom:
               secretKeyRef:
                 name: aws-kms-key
                 key: key-id
   ```

2. **Encrypt Secrets Using External Key**
   ```sh
   sealsecret --external-encryption-key <AWS_KMS_KEY_ARN> <secret-file.yaml> > sealed-secret.yaml
   ```

#### **Key Considerations**
- **Cross-Cluster Sealing:** Ensure the decryption key is accessible across all clusters.
- **Key Rotation:** Automate key rotation via your external provider.

### **3. GitOps Workflow (ArgoCD/Flux)**
Deploy `SealedSecret` alongside manifests in Git using **GitOps tools**.

#### **Example (ArgoCD Application)**
```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
spec:
  destination:
    server: https://kubernetes.default.svc
    namespace: my-app
  source:
    repoURL: https://github.com/myorg/myapp.git
    path: k8s/manifests
    targetRevision: HEAD
    helm:
      valueFiles:
        - values.yaml
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

#### **Schema Reference (GitOps)**
| File | Purpose |
|------|---------|
| `sealed-secret.yaml` | Encrypted `Secret` (committed to Git). |
| `values.yaml` | Contains references to the `Secret` (e.g., `envVars: [{"name":"PASSWORD","valueFrom":"SecretKeyRef": {"name":"my-secret","key":"password"}}]`). |

```sh

```

## **Best Practices**
✅ **Use Strict Mode** (`STRICT_MODE=true`) to prevent accidental data leaks.
✅ **Rotate Encryption Keys** periodically (via AWS KMS/Vault).
✅ **Avoid Storing Encrypted Secrets in Git** unintentionally (e.g., via `--dry-run` leaks).
✅ **Use RBAC** to restrict who can decrypt secrets.
✅ **Audit Logs** for decryption events.

---

## **Common Pitfalls & Fixes**

| Issue | Cause | Solution |
|-------|-------|----------|
| **Decryption Fails** | Key mismatch | Verify `--external-encryption-key` is correct. |
| **SealedSecret Not Found** | Wrong namespace | Use `-n <namespace>` flag. |
| **Permission Denied** | Missing RBAC | Grant `SealedSecret CRD` access to the controller. |
| **Secrets Leaked in Git** | Forget `kubeseal --dry-run` | Always test decryption locally first. |
